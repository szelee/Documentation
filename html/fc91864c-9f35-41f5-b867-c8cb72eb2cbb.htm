Remarks

BACKGROUND

This FEWS allows all FEAs to enquire as to whether a journey is possible at the times requested by the user, and provides details of the train times, route information and fares in response to the request. This FEWS supplements the previous Journey Planning Call 1 with all the possible fares information for each journey returned from IPTIS. The fares are provided for each passenger type included in the input – ie adult, child, railcard holder. If available supplements are also provided to relevant journeys and fares e.g. Car parking and on train shop vouchers.

The Journey Planning Call 2 is a simple request – response mechanism. The FEWS call can be made from all retail channel FEAs. The retail channel making the request is included in the request message.

Railway locations are defined using 3 character 'CRS' and 4 character 'NLC' codes, and the CBE uses these codes in the web service calls. There is a requirement that the FEA is able to translate to and from these codes such that meaningful location names are presented to the customer. The necessary reference data is made available to the FEA by the CBE through separate web service calls, see section 1.3.

CBE will provide details of the valid walk up fares for each journey opportunity returned in the FEWS call, but the front end must make it clear that other Advance Fares might be available that could be cheaper (see assumptions).

This document has been updated following CR100 Journey Planning enhancements requested by SC. This version also includes additional RTTI scheduled information in the output for a service where available. It also includes the PassengerGroupFare structure which may be returned by IPTIS if such fares are possible, this fixed an earlier defect CBE-7621.

PROCESS DESCRIPTION

This call from the FEA to the CBE results in a call to IPTIS for route, timetable information and fare information for each service returned. The fares include the range of fares for adult, child or railcard holder depending on the passenger details input. This call is typically but not always, preceded by a Journey Planning Call 1 where a particular time or route is identified. This will avoid large volumes of fares data being sent back in each response thus requiring the FEA to handle a greater amount of data, which may simply be discarded as the end user refines their journey search.c

The input may also specify a particular class – standard or first, in which case only the fares associated with this class are returned. If this input is not designated then the output returns all fares classes. In line with current practice, the applicability of railcard discounts is determined by IPTIS.

For Ticket Types requiring a mandatory reservation booking requirement, IPTIS will only return fares where there is reservation availability. Reservation Status is included per Journey. This is passed to the FEA by the CBE using the data from the IPTIS response and will be one of:

  • Mandatory (These are quota controlled fares.)
  • Recommended
  • Possible
  • Not possible.

When a valid Promotion Code is found and the Promotion Type equals “National”, the Promotion Code will be used in an IPTIS fares call as though it is a Railcard Code. IPTIS will return the correct passenger status codes and fares.

When a valid Promotion Code is found and the Promotion Type equals “Local”, the CBE will:

  • Return only the Ticket Types appropriate for the promotion as described by the promotions reference data.
  • Calculate the appropriate fare using the Discount Type and Discount Value in the Promotions Reference Table.

Where the destination location is London Terminals the FEA is required to check the London stations restrictions list depending on the origin station, so that the user is informed of these. The CBE provides this data feed to the FEA – see section 6. For example a person travelling from Slough to London Terminals would only be allowed to travel to Paddington, whereas someone travelling from Brighton would be allowed to travel to Victoria, Vauxhall, London Bridge, Blackfriars, City Thameslink, Farringdon and St Pancras (Thameslink). In both cases the route is defined as 'any permitted' and the fare destination location is 1072.

If both the origin and destination location NLC codes are within the London Travelcard area, (In-Boundary) then in addition to the P2P fares returned the call returns: Anytime Travelcard for zones 1-4. and zones 1-6 and Off-Peak Travelcard for zones 1-6.

The input also includes maximum journeys to allow the user to request a number of journeys to be returned, which cannot be greater than the CBE upper limit for maximum journeys.

Supplements are returned if they are available in IPTIS associated with the service or route and/or locations e.g. car parking at a designated station or on shop vouchers on VT services. The supplement code is returned and this is looked up in a supplements reference table to provide the name of the supplement and to get the sundry code associated with it.

The output has been extended to includes Train Seating Class/ Train Sleeping Class details, information relating to disruptions affecting journeys returned. Train facilities are returned where available in IPTIS for a journey and also overtaken trains are indicated.

If a train splits the RSID field returned gives this information.

The call includes support for accompanied adults and children. The CBE returns these passenger types obtained from IPTIS and includes these where available (see PassengerGroupFare structure in section 5.2). In addition a new optional field fare type is provided in the output where it applies to a travelcard.

Fare Signature is a string that is returned within IPTIS Journey Planning response for each fare option. The fare signature contains the following information:

  • Ticket FTOT
  • Fare Setter
  • Route Code
  • Restriction Code
  • Fare Origin
  • Fare Destination
e.g. ‘SOS|IWC|00000|2C|0438|1072’

This version supports the inclusion of RTTI information for each leg of train services and RTTI scheduled information where available from NRE and also zonal fares.

As part of this release of FEWS 21.0 a new element called upgrade details has been added to the output. This includes data not from IPTIS, but from locally held and maintained tables within the CBE. No example is available currently, but the structure is included for completeness. This data is only available to the web retail channel for VTEC.

When split ticketing (ST) discounted fares are requested (via the new DoSplitTicketing flag added to the input request), CBE will attempt to provide discounts for appropriate non-split fares, by offering a price match with the corresponding split fare. For each journey in the response, IPTIS will provide the following ST fares:

  • Cheapest standard advance fare
  • Cheapest standard flexible fare (single direction)
  • Cheapest standard flexible return fare
  • Cheapest first advance fare
  • Cheapest first flexible fare (single direction)
  • Cheapest first flexible return fare
The cheapest advance split fare should be used to discount the relevant non-split advance fare where the fare setter for the non-split fare is IEC. The cheapest single direction flexible split fare should be used to discount the least flexible non-split single direction flexible fare where the fare setter for the non-split fare is IEC. The cheapest return flexible split fare should be used to discount the least flexible relevant non-split return flexible fare where the fare setter for the non-split fare is IEC

Split fare discounts applicable for a given journey will be returned within the Journey Fare Legalities collection for the journey. For the relevant non-split fare being discounted, the Legality element will include a PricingDetails element which gives details of the ST discount. This element will be populated as follows:

  • Discount Id : Not populated
  • Discount Name : ‘SplitTicket’
  • DiscountDescription : ‘SplitTicket’
  • Discount Rate Type: Fixed
  • Discount Rate: Not populated
  • Total Discounted Fare (For all passengers on journey) : Total ST discount
  • Passenger discount details (for each discounted fare):
    • DiscountStatusCode : Not populated
    • No. of Passengers
    • Discounted Fare: Individual ST discounted fare
    • Discounted Fare Hash : A hashed value calculated using the Discounted Fare, the fare Signature of the fare being discounted, and a private cryptographic salt value held within CBE and updated each day.
    • RailcardCode*
    • OriginalFare : Non-discounted fare (that also appears in FareOption element)
    • OriginalStatusCode : Not populated

Such ST discounts will only be offered if they provide a cheaper fare than the best sales offering discount available for the relevant journey fare, if any.

Alternatively, ST discounts should allow sales offering discounts to be applied on top, so potentially 2 PricingDetails element details would be returned for each journey / fare option : one detailing the ST discount available, the other detailing the sales offering discount to be applied on top. This alternative solution should be ignored until confirmed by LNER.

DoSplitTicketing is only supported on the Web and Call Centre channels.

If supplements are not requested – the Journey Supplement Validities and Supplements elements will be returned as empty.

Where a fare is returned with validity code 19 (no seat available) or 21 (NRS not available):

  • If Fare Category is Advance
    • If Validity Code = 19 (No seat available)

      Return the fare and the status as this signals that a no seat reservation is possible

      <FareStatusCode>19</FareStatusCode>
      <FareStatusDescription>No seat available</FareStatusDescription>

    • If Validity Code = 21 (NRS not available)

      Do not return the fare as a seat cannot currently be booked

  • Otherwise return the fare and an NRS status against the fare of either:

    <FareStatusCode>19</FareStatusCode>
    <FareStatusDescription>No seat available</FareStatusDescription>

    Or

    <FareStatusCode>21</FareStatusCode>
    <FareStatusDescription>NRS not available</FareStatusDescription>

If original transaction and its journey are found and is coj eligible, then:

  • Provided is only journey if:
    • Journey departure time is still more than the minimum lead time in the future
    • Journey and ticket type is fulfillable by Eticket
  • Provided is only Fare option if:
    • Target Ticket type is listed in coj eligible tickets types of original ticket type in Admin UI
  • CojAdminFee are calculated for each passenger fare or for passenger group fares (whichever is applicable) (or 0 if WaiveAdminFee is set to true)
  • CojSundryCode is present in response.

When FEWS 2.1 is called to obtain COJ journey and fare options, if the customer is entitled to have delta fare waived (ie their original journey is non-viable due to either timetable or reservation changes and the target journeys in the response are within the time window configured in Admin UI) then:

  1. For each journey provided in the response, determine if there is a fare legality for which TotalCojCosts.CojDeltaFare is 0, which does not have a Legality Code of 25 (no availability), and for which the associated fare option has the same class (standard or first) and category (advance, offPeak, anytime, etc) as the original journey fare. If so no further action is required.
  2. Otherwise, determine if there is at least one other fare legality which does not have a Legality Code of 25, and for which the associated fare option has the same class (standard or first) and category (advance, offPeak, anytime, etc) as the original journey fare. If so, select the fare legality which has the lowest value for TotalCojCosts.CojDeltaFare and set this value to 0.

ASSUMPTIONS

  1. For Mobile channels, the capability of the receiving device will be a function of the Web Server and not the CBE.
  2. It is assumed that responsibility for continuity of services displayed over successive “Earlier” or “Later” timebands requests lies with the Front End Application.
  3. Where the destination location is London Terminals the front end application is required to check the London stations restrictions list depending on the origin station, so that the user is informed of these.
  4. The FEA will decide which supplements are to be offered to the customer by retail channel and ignore those not required. The CBE will return all that are available from IPTIS.
  5. It is assumed that the FEA will advise the customer that cheaper fares may be available at other times by means of message e.g. “Other fares may be available that may be cheaper if you choose to travel at a different time and / or date”.
  6. It is assumed that the FEA will have retrieved the reference data to manage the initial front end validation by means of FEWS 10.n series.
  7. The FEA will have the list of railcards/promotions which are to be allowed for each retail channel as determined by the TOC or RSP as appropriate. This will not be validated by the CBE.
  8. For VTEC Web channel only: It is assumed that the FEA will manage restricted days for Extended Horizon where there is an intention to release the ticket quota in the extended horizon but it has not been done yet. The customer may be asked if they would like to subscribe to be notified when quota is made available.
  9. For VTEC Web channel only: It is assumed that the FEA will manage Excluded days for Extended Horizon where there is no intention to release the ticket quota for the extended horizon, eg. Bank Holidays

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. Parameters are correctly formatted.
  3. If CojOriginalJourney is provided and GetReturnFares is true, CojOriginalJourney must provide InboundJourney.
  4. Station Locations are valid.
  5. All stations in input must be different locations.
  6. Outbound Date < n weeks from current date (n = booking window currently 13 weeks).
  7. Travel (Outbound) date+ time < Return date + time where there is a return leg.
  8. Number of adults and children combined > 0 and < n where n is configurable:
    • 1 - 9 for web
    • 1 - 9 for mobile
    • 1 – 69 for call centre, ticket office, on board
  9. Where promotion code exists in the request:
    1. It must exist in the Promotions Reference table
    2. It is valid for the effective “From” and “To” sale and journey dates in the table
    3. It is valid against the number and type of passengers input
    4. It is valid for the customer (Promotion Target + Customer type) where applicable.
  10. The carnet value must be one of the following values 0, 5, 10 (this is a configurable list).
  11. Where the carnet value is not zero the passenger number either adult or child must be = 1.
  12. Travel date is not in the past.
  13. Railcard must be valid for retail channel. There is no limit on railcards other than cannot be more than number of passengers
  14. Outbound time can be empty. CBE will use current date time in this case. Outbound time can also be in the past (configurable time period as in the FEWS 2.0)
  15. Maximum journeys should be < CBE configured limit, whichever is smaller and should always be > 0.
  16. If Retail Channel is Call Centre or ticket office or on board then Operator ID must be valid and logged on with the correct access role. If retail channel is web/mobile or TVM then sales point details must be present. Check that only one of user credentials and sales point details is present.
  17. Max No of Changes must be numeric of (0-9) where 9 means “Any” – if not provided the default value of 9 will be used
  18. If AvoidStationLocation provided but type not specified – default value of “notvia” will be used
  19. If EnforceRouteVia provided but type not specified – default value of “passing” will be used
  20. If EnforceRouteVia type=”Route” then only one single 5 digit route code can be specified.
  21. If EnforceRouteVia type = changeat or callat then either (CRS and or NLC) must be provided
  22. Service Provider must be a valid two character service provider (as returned in FEWS 10.5 Get Service Providers).
  23. If ShowOvertaken enumerator not provided, default to include
  24. If ShowLiveRunning Flag not provided, default to false
  25. If ReduceLondonTransferTime Flag not provided, default to false
  26. If InterchangeTimeIncrease provided value must be an integer between 1 and 120
  27. CheckReservations Flag is False if RC = On Board, else default to TRUE
  28. InterchangeTimeIncrease and ReduceLondonTransferTime are mutually exclusive – i.e. both cannot be specified in the same input request
  29. If ShowZonalFares flag not provided, default to false
  30. Travel Date must be less than or equal to the Extended Horizon End Date
  31. Travel Date must not be an Excluded or Restricted date for the Extended Horizon
  32. Origin and Destination must be a valid pair for the Extended Horizon
  33. The PRM_ENABLE_SALES_OFFERING (Config UI) must be set to TRUE
  34. The sales offering Code is valid and current
  35. If Service Provider Filter is provided but type is not specified, default value is AtLeastOne
  36. If ShowRepairedJourneys Flag not provided, default to false
  37. If DoSplitTicketing flag is not provided, default to false. Ignore unless channel is web or call centre
  38. If GetSupplements flag is not provided, default to false
  39. Where CojOriginalJourney item exists in the request:
    1. Its TransactionNumber and JourneyNumber also need to be present
    2. Sales Transaction and ticket need to exist
    3. Ticket type of original transaction needs to be eligible for change of journey
    4. If both OutboundJourney and InboundJourney were provided in CojOriginalJourney, these journeys must be paired with OutboundInbound pairing type, or at least have reversibly same origin and destination stations and ‘Accept any basket item pairing' flag configured in Config UI set to true.

The following 3 additional business validation checks only apply to the web retail channel for VTEC, where the CBE local table of extended horizon dates is available, otherwise should be ignored.

  1. Travel Date must be less than or equal to the Extended Horizon End Date
  2. Travel Date must not be an Excluded or Restricted date for the Extended Horizon
  3. Origin and Destination must be a valid pair for the Extended Horizon
