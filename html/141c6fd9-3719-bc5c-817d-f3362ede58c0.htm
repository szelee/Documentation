Remarks

BACKGROUND

This FEWS supports the creation of a user account. It captures all the key customer information required in order to complete a sale and correctly fulfil the ticket purchased. In summary it contains the following logical groups of data:

  • Name and address and any delivery addresses where different and contact details (email tel. nos etc)
  • Personal and security details required for logon and for customer support (passwords Photocard id, DOB, etc)
  • Loyalty Details (if applicable)
  • Opt in flags for up to 3 marketing schemes as required
  • Seating preferences (when reservations required)
  • Favourite journey details.

The creation of a customer account in the database results in the creation and the output of a customer account reference. This is provided to the FEA but is not expected to be known or disclosed to the customer. The customer email account details are used as user names for web logon.

This version has been updated to include enhancements for holding Loyalty details (CR95) and enhanced customer account information (CR99)

PROCESS DESCRIPTION

This case creates the customer account record in the CBE database. A customer may register several times and have different and/or similar accounts where different email addresses are used. This is not prevented and no cleansing is done. On creation of a new customer account a customer reference is assigned and returned to the FEA. The Customer Id is generated by the CBE and is for internal use, passed to front end to indicate successful account creation. This customer reference is used elsewhere in later FEWS.

Included in the data items are Customer opt in flags – which were originally included to meet Virgin’s specific customer loyalty requirements. These have been made generic: Customer Opt In Flag 1 and Customer Opt In Flag 2, so they can be used in the future for any marketing initiatives.

The data items include a number of optional items - season ticket details (photo card id) and favourite journey details - for future use. Favourite journey details are captured and held in the customer record, but is not used by the CBE but is present for the FEA to use to pre populate a specific journey planning call.

This FEWS has been extended to include Loyalty details for VTEC (CR95), this allows the customer to register their Loyalty details: their scheme and card number. Currently only Loyalty and Virgin Atlantic Flying Club schemes are supported.

CR99 has also introduced major changes to this FEWS to enhance the customer account details to include:

  • restructured address details for different uses
  • optional password hint
  • contact preferences
  • improved railcard details with expiry dates
  • improvements to favourite journeys
  • opt in flag for marketing

ASSUMPTIONS

  1. It is assumed that the Front end will perform PAF validation of the address – this will not be the responsibility of the CBE.
  2. It is the responsibility of the FEA to enforce the default value of the direct service flag to ‘all services’ as input to the CBE, if the customer does not respond to this field as this is mandatory.
  3. It is the responsibility of the FEA to retain the customer details while the customer session is open so that this data can be reused in subsequent FEWS.
  4. It is assumed that the FEA for SWT will suppress the Loyalty details at the present time as this is only required for VTEC.
  5. It is assumed that the FEA will verify the email address by asking the customer to submit it again to check it matches (CR099 requirement)
  6. It is assumed that the FEA will verify the password by asking the customer to submit it again to check it matches (CR099 requirement)
  7. The FEA is required to use the password prompt associated an email address if present in the customer account details if the customer cannot remember their password. (CR099 requirement).
  8. It is assumed that the FEA will not provide outbound nor return journey details in favourite journey where journey type is Season.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. Transform email address into lower case, check it is unique and store against customer account.
  3. If Ticket Class is present values should be should be First, Standard (nulls=All).

  4. Customer Address validation:

    1. Must consist of Postcode and Address Line 1 and 2 for all countries
    2. Postcode should be 4 to 10 alpha numeric characters
    3. For USA addresses line 4 (City), Line 5 (State) and Postcode (ZIP) must not be blank.
  5. Age of customer must be => than 16 years.
  6. If Ticket class is present in favourite journeys values must be First or Standard.

  7. Check photocard ID:

    1. is unique and not already associated with another customer account
    2. is associated with a smartcard if smartcard details are present
    3. is associated with a railcard if railcard (Y-P) details are present
    4. is correct format
  8. Terms and conditions agreed.
  9. Seating preferences if present exist.

  10. Check Railcard

    1. if present it exists see FEWS 10.6
    2. expiry date has not passed
  11. Type of Ticket exists.
  12. If Loyalty card details group is present then all fields within must be present.
  13. If present Check Loyalty scheme name is a valid one (Nectar and Virgin Atlantic Flying Club)
  14. Number of loyalty cards being added should be within max limit configured against the scheme (to be advised
  15. Loyalty card should not be expired
  16. Loyalty details should never be present where Retail channel is SWT.
  17. Check Phone number is present if contact preferences field states Phone (not mobile)
  18. Check security details if present do not exceed 3
  19. Journey name should be unique to the customer account
  20. If Journey type (in favourite journeys) is Season then outbound and return details should be absent.
  21. If Journey type (in favourite journeys) is P2P then outbound details must be present.
