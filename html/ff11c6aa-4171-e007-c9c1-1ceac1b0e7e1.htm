Remarks

BACKGROUND

This FEWS request returns a summary list of customer sales transaction history by purchase date or by journey departure date. The response includes summary information to allow identification of an individual sales transaction to support a subsequent detailed Sales History enquiry (FEWS 4.3.1). The FEWS applies to all channels as required. This document has been updated to include change request additions to include car parking purchases, evoucher purchases and Loyalty details as applicable.

BUSINESS PROCESS CONTEXT

The diagram below shows the context of where in the business process this FEWS call occurs.

PROCESS DIAGRAM

PROCESS DESCRIPTION

This FEWS allows the customer or operator, to request their sales history. The request may be sales for one of the following:

  • For a period of up to n days from the current date (default value configurable by retail channel)
  • For a designated journey departure date
  • For a sales transaction date.

If no search criteria are provided by the FEA then the CBE retrieves the summary sales history from the current date to the above configurable date value.

The FEWS allows the input of pagination criteria, to manage the output where there is a large amount of sales history in the time period. The pagination is optionally controlled by 2 input fields; the number of pages and the page size. Where this data is not provided the number of pages is limited to one, and the size is set to 20 items per page. These default values serve to limit the data returned to the FEA regardless of the amount of data returned from the search of the database.

The CBE will use the sales transaction date, (the booking date) to provide history over the requested range (Number of History Days). Alternatively where the search is by journey departure date, this will assist in providing support where the customer rings up with problems with ticket for travel for today and cannot recall when it was purchased.

The configurable item which restricts the amount of history returned by this FEWS is set by retail channel so that the web and mobile channels can restrict the amount of history to 90 days. For the call centre (and ticket office) it is recommended that this should be set to 400 days to allow the operator to view more historical data to fully support the customer.

The output is one or more items of summarized sales transactions so that the customer or operator can select which item they require to drill down into to see more details. This is then done by a subsequent FEWS 4.3 (for web or mobile or TVM) and FEWS 4.3.1 (for Call Centre). Where the request is from an unattended retail channel, the output shows only sales transactions which are in a sales state of confirmed. Where the request has come from an attended retail channel the output includes sales with a sales status of “Awaiting Authorisation” as well as confirmed sales. This will assist an operator to troubleshoot problems as required.

The output is paginated to manage the potential volume of data which could be returned from the CBE.

The output includes summary refund details, which include the refund application state. The refund states are new, in progress, approved, rejected, completed, refund failed.

The sales history output has been updated to include details of any car parking purchases, evouchers and also the Loyalty points gained for the sales.

When eVouchers are included in the output, the Code is masked by the CBE showing only the first five and last five characters

Transactions can be filtered based on transaction pairing type (eg. LinkedBikeTransaction, ChangeOfJourney,...) and exclusion transaction type (eg. Original, Paired) both placed in TransactionPairingsToExclude collection within a request.

ASSUMPTIONS

  1. Prior to this call, FEA will have obtained the Customer reference through Customer Account Sign-on, or previous Call Centre Operator or Ticket Office processes.
  2. It is assumed that for the call centre operator, they have their role access permissions set to allow read of sales history data.
  3. It is assumed that the FEA will reduce this output data down to a suitable level as required.
  4. Where the full history period is not available due to the default settings which limit the amount of history it is assumed that the FEA will manage the customer expectations.

VALIDATION

Parameter Validation:

  1. Format of Date fields

Business Validation:

  1. Customer ID exists within the CBE Database
  2. History date from (start date) must be greater than (current date – default value of number of history days).
