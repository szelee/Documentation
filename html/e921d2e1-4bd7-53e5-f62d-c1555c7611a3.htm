CbeWebServiceGetSalesTransactionDetails Method

This is a request which will retrieve the details of a specific customer sales transaction. This call includes details for all purchased items in a single sales transaction including:

  • Season and Point to Point tickets
  • Reissued season ticket
  • Car parking
  • Reservation purchases and details of these
  • eVouchers
  • Railcards
  • other associated items
  • payment details for the purchase
  • fulfilment status details

Remarks

BACKGROUND

This FEWS allows a Call Centre operator or customer to request the details of a specific customer sales transaction. This call returns not only the sales transaction details but also fulfilment status details from the CBE database. In this way the operator can provide customer support in case of issues arising with ticket collection or sales. This will be used in the case of a customer enquiry or for the processing of ticket refunds. The same call is used for web customers who wish to see the details of sales transaction, when a subset of this data returned can be used. (See assumptions).

PROCESS DESCRIPTION

This is the call which typically follows the selection of a specific sales transaction from the summary list provided by FEWS 4.2. The key input to this FEWS 4.3.1 is the sales transaction reference, which typically has been ‘selected’ from FEWS 4.2. This FEWS 4.3.1 can also be called independently of FEWS 4.2 where the required input data is already available to the FEA.

This request will retrieve the details of a specific customer sales transaction. This call includes not only sales transaction details for both purchased tickets but also includes reissued season ticket details, railcard purchases and evoucher purchases. It also includes fulfilment status details so that the operator can provide optimum customer support in case of issues arising.

The output from the CBE includes:

  • The customer id and name
  • all the details of the journeys in the sales transaction
  • associated reservations
  • the fulfilment status of the sales transaction
  • the sales transaction status
  • any evoucher sales associated with the transaction
  • the payment details
  • any refunds associated with the sale
  • any railcard purchases
  • any replacements/reissues for season tickets etc.
  • Loyalty card points associated with the sale

This data is provided so that the operator has all the necessary details to give customer support as required.

The sales transaction status where MOP is card (value =X), is output to show whether it is:

  • Awaiting Authorization (awaiting confirmation by the PSP)
  • Failed (failed to be authorised by the PSP)
  • Confirmed (authorised by the PSP)

In the first two sales states, the sale will not progress to completion so there will be no fulfilment history details to output. In the last case the sale is complete so the fulfilment activities will have been triggered by the CBE and fulfilment history details for smart or CCST will be present.

Where a refund reference number is present this will only occur where a refund has been requested for the original sale. There may be more than one in the case of a season ticket with multiple Delay repay refund requests raised. The presence of a refund reference number can be used as the link to FEWS 9.6 to review the refund details as required.

This version has been updated to include fulfilment details associated with a sale transaction with Mticket as the fulfilment option. The mticket fulfilment details include the various state changes corresponding to the Mticket lifecycle. This is change is associated with CR071 for dynamic MTicketing.

In addition this version also includes updates relating to products sold and managed by the Admin UI function for pricing. Promotion Discount Details are now included and are returned for each journey (currently P2P only) where present. These extra fields should only be populated when the passenger fare status is 018 or 019.

If an additional discount is applied with railcard, DiscountAppliedWithRailcardCode will be set to 3 character Railcard Code.

Fare Signature is a string that is returned within IPTIS Journey Planning response for each fare option. The fare signature contains the following information:

  • Ticket FTOT
  • Fare Setter
  • Route Code
  • Restriction Code
  • Fare Origin
  • Fare Destination
e.g. ‘SOS|IWC|00000|2C|0438|1072’

In sales transactions for customers that are opted in Loyalty scheme and basket contains items eligible for loyalty, response will contain CashbackRewardValue and may also contain CashbackCharityId (on GetSalesTransactionDetailsResponse level) for every eligible SalesTransactionJourney or SeasonTicketDetails.

For each sales transaction with which the current transaction is paired (eg non-issued transaction and re-issue transaction in the case of lost in post, or original journey transaction and replacement journey transaction in the case of COJ), The Transaction Pairings collection should contain an entry identifying the transaction reference number for each transaction in the pair (including the current transaction), the relevant basket item from each transction if appropriate, and the type of pairing). Note that there may therefore be more than one entry in the collection for the same pair of transactions.

Status flag is identifying whether the ticket was processed by TT changes and some timing data was updated (Updated) or journey was deemed non-viable (Cancelled). If ticket was not processed by any TT changes, Status is empty (null).

At each point of TT horizon processing, TimetableNotificationOutcome should be present in SalesTransactionJourneyDirection. When journey was still viable, HistoricalJourneySignature will hold information about journey signature before the changes made at this date. If journey was deemed non-viable, there should be only one entry of TimetableNotificationOutcome with reason listed from below options:

  • ServicesAreRemoved
  • JourneyDepartureNotWithinAcceptableThreshold
  • JourneyArrivalNotWithinAcceptableThreshold
  • BoardingRemoved
  • AlightRemoved
  • CannotMakeTransferBetweenStations
  • CannotMakeChangeAtStation

ASSUMPTIONS

  1. Prior to this call, FEA will have obtained the sales transaction number through a customer sales history summary request (FEWS 4.2).
  2. It is assumed that the sales transaction value (which is for rail retailed products only), will exclude non rail products bought at the same time, however the authorization amount may be greater than this value where non rail products have been purchased at the same time.
  3. As this FEWS will now be used for Web retail channel instead of FEWS 4.3 it is assumed that the FEA will filter out data that is not required to be displayed to the customer.

VALIDATION

Parameter Validation:

  1. All mandatory fields are present
  2. One of the input parameters must be present
  3. Invalid parameter format
  4. Invalid string length

Business Validation:

  1. The Sales Transaction exists
  2. Operator (user credentials) are valid
  3. Operator (user) privileges allow access to customer sales
