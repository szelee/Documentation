CbeWebServiceReservationBooking2 Method

This call triggers a reservation request to NRS. Optionally, reservation request can contain additional reservable items - bike places, seating preferences or requirement for specific seat in the train. The FEWS supports, public facing, unattended TIS such as Web TVM or Mobile retail channels as well as attended TIS such as Call Centre or Ticket Office retail channels.

Remarks

BACKGROUND

This call triggers a reservation request to NRS for either:

  • an unattended TIS (for the web and mobile and TVM channels). It allows the customer to request a reservation for a single journey, or for both outbound and inbound journeys as applicable. The customer can also choose what type of reservation is required. The choice of reservation properties is limited by RSP to unattended TIS to - seat only, and the position and direction of the seat and other associated accommodation attributes.
  • an attended TIS (for requests from trained operators). It allows an operator/user in a Call Centre or Ticket Office, to make a reservation in NRS for one or more reservable items on a particular train service. The call typically follows an availability request to NRS for the particular reservable items (FEWS 8.0.1). This call triggers a reservation request from the CBE to NRS for the reservable items requested including seats, bike places on train and more precise seating preferences eg by luggage rack. This call is equivalent to that shown above but includes additional reservable items only available through these retail channels as a controlled NRS user.

PROCESS DESCRIPTION

This call allows the customer to request a reservation for a journey for single, outbound and or inbound legs (for a return journey). Where the journey has multiple legs a reservation of each leg is sought by the CBE from NRS. Where reservations are not available for some or all legs no reservation details are returned. Where reservations are available for some or all legs these are returned for each leg of the journey.

The customer is also allowed to select additional accommodation attributes (in case of the web/mobile channels these are restricted to those associated to seat). The attributes supported by the web/mobile channels are:

  • Quiet coach
  • Seat position - Window or aisle seat
  • Table seat
  • Power socket.

The above attributes are not necessarily available on all trains. On some train services seats which are both window and aisle may exist (eg on a Virgin Trains Pendolino). The seat position data returned by NRS to the CBE depends on the way in which the seats have been set up in NRS by the operating TOC.

The CBE takes this request and requests a reservation to meet the requirements from NRS. NRS returns one or more reservations for each passenger which may or may not meet the accommodation attributes requested. The output returned from NRS to meet the requirements is controlled by NRS algorithms and not by the CBE.

For multiple passengers, reservations returned may or may not be together as it is not permitted to set the together flag from unattended TIS sales requests (RSP policy). Similarly it is not possible to request other reservation facilities eg bike reservations as these are limited by RSP to attended TIS only.

The reservation details for each leg are returned by the CBE to the FEA, for each leg identified by the train RSID, the coach and seat number. In cases where NRS is not able to reserve an actual place on a service (e.g. due to Overbooking or Counted Places) the Coach Identifier and Seat Number are returned as “0” (zero). This is not a fault but a feature of NRS.

Where a service may be a split train as identified by more than one RSID provided in the input e.g. Rsid>EM115603;KET;EM115601</Rsid> in this case the CBE takes the last element split by the ; to request reservation in NRS.

This call allows for a customer to enter their seat picking details - having obtained this from previous calls (using FEWS 8.4 and 8.5) and a graphical display of availability of seats in coaches provided by the FEA. The response should return the exact seat pick details if still available otherwise it will return seat details as close to that requested.

For attended TIS (call centre etc) there are more attributes supported which can be requested by trained users these include:

  • Near luggage rack
  • Near WC
  • Priority seat

In this version a new field the ReservationPrepurchaseIndicator is required in the input to show whether the reservation booking is being made in advance of a purchase (ie =TRUE). In most cases the call will be associated with a sale so the default is set to FALSE. The CBE must record this detail against the customer’s reservation record and a date and time stamp held for this reservation booking. In this case no sales transaction will be associated with it. These reservations are cleared down and cancelled in NRS after n hours (where n is configurable and to be agreed with RSP/SC). (This has been added as part of CR117 amend reservation part 2).

In addition where the travel dates provided in the input, are beyond the standard booking dates, the CBE will validate if Extended Horizon is enabled for the TOC Retail channel before carrying out the business validation. If Extended Horizon configuration is allowed then the station location pair list will be a list of station pairs allowed. If not allowed then then the station pair list will be a list of locations not allowed fo extended horizon journey planning.

ASSUMPTIONS

  1. It is assumed that a trained operator will refer requests for bike reservations for other TOCs services which cannot be handled by this call to the individual TOC customer support centres; (where a reservation is required to be done by a different method).
  2. It is assumed that the FEA will hold the reservation details for the customer’s journey and only provide these to the customer once the sale is confirmed by receipt of a payment. It is not normal practice to display these details until the sale is completed.
  3. It is assumed that the FEAs have already provided the seat layout details previously to allow the passenger to select a coach / seat to use as input in this call.
  4. The FEA must ensure that the ReservationPrepurchaseIndicator is correctly set to distinguish between a pre purchase reservation which is being saved for a later ticket purchase and the more common case where is being associated with a ticket purchase (FEWS 6.1).
  5. It is assumed that the FEA will manage restricted days for Extended Horizon where there is an intention to release the ticket quota in the extended horizon but it has not been done yet. Customer may be asked if they would like to subscribe to be notified when quota is made available.
  6. It is assumed that the FEA will manage Excludeed days for Extended Horizon where there is no intention to release the ticket quota for the extended horizon, eg. Bank Holidays

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

  1. Date and time is valid
  2. Coach reference must be alpha
  3. Seat reference must be numeric
  4. Number of passengers must be less than 9
  5. Either ReservationPreferences or SeatPickDetails must be provide for passenger group
  6. All mandatory fields exist in input request
  7. User credentials / Sales Point Details are valid
  8. Check if the operator / user credentials have permission for this call (see latest version of FEWS Permissions)
  9. Origin and Destination locations exist
  10. Date exceeds the current booking window
  11. Accommodation Type if present must valid. (SEAT)
  12. Accommodation Attribute must be valid (WIND, POWE, …see RSD data)
  13. Number of passengers must 9 or less
  14. If RC = Web/TVM/Mobile, then sales point details must be present
  15. If RC = Call Centre/Ticket Office, then user credentials must be present
  16. If Seat Pick Details are present, then Coach and Seat references must be present, exist for the specified RSID and must not be already booked. Information about Coach and Seat references can be obtained from FEWS 8.4 and 8.5
  17. If Seat Pick Details are present, then the number of specified seats must match the number of passengers
  18. Seat pick details should only be present for the web/mobile
  19. If RC = web/mobile (unattended TIS) then facilities should be absent
  20. The attributes requested by RC = web/mobile/TVM (unattended TIS) should be valid against the list defined by RSP for unattended TIS
  21. The attributes requested by RC = call centre/ticket office (attended TIS) should be valid against the larger list for attended TIS
  22. Origin and destination locations cannot be an IPTIS group location
  23. Travel Date must be less than or equal to the Extended Horizon End Date
  24. Travel Date must not be an Excluded or Restricted date for the Extended Horizon
  25. Origin and Destination must be a valid pair for the Extended HorizonCheck using NRS configuration that this NRS message type has been disabled for this channel
  26. Povide AssistanceTypeAttribute is allowed only for TOC that is specified in config UI
  27. Max wheelchair spaces can not be exceeded. Max value is defined in config UI
  28. Max wheelchair companions could not be exceeded. Max value is defined in config UI
