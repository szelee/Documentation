Remarks

BACKGROUND

This FEWS enables the Operator to update the Refund Request database with the results of a manual review of a refund claim. This may be an authorisation or rejection. The refund should be in the state of CalculationComplete.

This call should be preceded by a lock on the refund record (FEWS9.1) and unlocks it on completion of the response. This document has been updated to include refunds of car parking and ETicket refunds.

BUSINESS PROCESS CONTEXT

The diagram below shows the context of where in the Refunds (Process Application) process this FEWS call occurs.

PROCESS DIAGRAM

Not required.

PROCESS DESCRIPTION

This FEWS enables the Refund Administrator or operator to update the Refund Request database with the results of a manual review of a refund claim. This may be an authorisation, rejection or review note.

The refund request may just be viewed by an administrator where no update is required. In either case this FEWS unlocks the refund record in the refund workflow queue.

The refund will be in the state of CalculationComplete, following the review the status of the refund is updated to:

  1. In Progress - If Review decision is not populated
  2. Approved - One or more ticket is approved and Refund Amount > 0 and all other tickets are rejected.
  3. Rejected - Whole refund rejected or Refund Amount <= 0

The (overall) refund status of the application is based on the sum of all the ticket refund states according to the agreed table.

Once the operator has made the refund decision so the status is not ‘In progress’ the CBE sends a suitable email informing the customer of the refund review outcome (only in case when customer email address is available). If the Refund Amount (calculated as Sum of Ticket Refund Amounts – Admin Charge) is less than or equal to 0, the CBE automatically sets the Refund Status as Rejected with refund decision code set to 999. If the ticket refund is authorised, for journeys with a live Reservation, i.e. the date has not expired, then the CBE cancels the reservation by a message to NRS.

The Admin Charge value is a CBE configurable item, currently set to £10. The operator does not need to know it, they just decide by setting a flag, whether it is applied or not.

For refunds of tickets on smart media, the smart cancellation request flag is required for triggering a smart cancellation/void message to the HOPS system.. Once the success message has been received by the CBE (asynchronously) from the HOPS, the fulfilment status is updated. On the next review date by the operator, they will see that the smart cancel/void request has been successful (as shown in FEWS 4.2) so that this refund/ticket can be ready to authorise.

Where a ticket has been sold with fulfilment by TOD the ticket number provided in the input should be the number printed on the coupon, and not the ticket number in the sales record (if sold by the CBE) when CTR placement was done by the CBE

For refunds of a car parking ticket the operator is required to calculate the refund amount and provide this in the input. In this case no refund reason code should be present. The output then returns the total refund amount including any car parking refund in the figure.

In the case of ETickets (MTickets), on successful completion of the FEWS 9.2 process and the refund value being returned, the CBE then updates the ETV Database deny list, so that the ticket is logically cancelled. In this way the ticket cannot be used again if it is presented at a gate.

Where a Total Discounted Refund Amount is specified, this will be the amount to be refunded to the customer, whereas the Total Refund Amount will be the amount to settle in Lennon.

ASSUMPTIONS

  1. In case of uncollected ToD tickets the Virtual issue will be automatically triggered and all tickets will be issued. It is assumed that where the refund request is not for all tickets in the CTR then this process is not used, and those tickets required for refund must be collected first and submitted before the refund is requested.
  2. In case of Smart fulfilment the fulfilment status will need to be checked as this will determine the type of message request (cancel or void) to be sent to the HOPS.
  3. It is assumed that the email showing the review details will be sent to the email address in the registered customer account details.
  4. In case of CCST or ToD if the tickets coupons haven’t been delivered yet the operator can’t process the request.
  5. It is assumed for smart usage the operator will check with the HOPS team to check this data manually in order to calculate the refund amount, as no such message currently exists in RSPS3002. This will need to be defined as a manual business process/future CR.
  6. It is assumed that the (overall) refund status of the application is based on the sum of all the ticket refund states according to the agreed table (Multiple refund states table)
  7. The FEA should not display refund type code 999 as this is not permitted by RSP.
  8. The FEA should indicate that where a negative refund value arises the refund is rejected.
  9. The trained operator should only enter the ticket number from the printed coupon, where the ticket has been returned. This is important where the ticket has been previously collected by TOD.
  10. It is assumed that the operator has correct access rights to approve overridden refunds.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

Parameter Validation:

  1. Mandatory fields must be present.
  2. When IsSmartCancellationRequested is true, ReviewDecision must be missing.
  3. There cannot be both Passenger Fare Ids and Ticket Numbers presented.

Business Validation:

  1. Login credentials must be valid.
  2. Refund Number must exist
  3. The Ticket Refund state prior to this update is either ‘ CalculationComplete or CalculationOverriden.
  4. The valid input values for this call for Ticket Refund Status are one of Approved/CalculationFailed
  5. Refund reason code valid based on the table of refund reason code values.
  6. Reason for Refund (SDCI+) code valid based on the table of SDCI codes
  7. If sale is by smart fulfilment Smart cancellation request flag must be present otherwise must be false.
  8. If smart cancellation request flag is true then validate that a smart cancellation/void request has not already been sent to the HOPS.
  9. If smart cancellation request flag is false then Review Decision must be present and validate that a smart cancellation/void request has already been sent to the HOPS.
  10. For tickets not fulfilled by TOD the Ticket Refund Details collection must contain exactly the same ticket numbers as requested in the refund request or modified by operator,
  11. If the ticket has been collected by TOD do not check the ticket number against the sales transaction record details. But if it is not TOD then confirm that the ticket number is present and can be found in the sales transaction.
  12. Confirm that where the TODCollected flag = FALSE the number of passenger fare ids present matches the number of tickets in the CTR. (Prior to virtual issue of CTR need to check that the numbers of items for refund are the same as number in the CTR. (see note 11 below).
  13. Check that the number of tickets or number of Passenger Fare Ids are < or = total number of tickets in associated sales transaction.
  14. Check that the number of refund tickets is equal to number of requested tickets.
  15. If the ticket is a TOD one and has been collected/issued by a foreign TIS check that all the TOD issue details (as per FEWS 9.1.2 input) are available against the refund record so that the correct BS record can be written for the refund. (This does not apply where the TOD tickets have not been collected).
  16. Check if record has already been locked for another operator
  17. Refund value cannot be absent in refund record (not been previously calculated).
  18. Check the operator/user credentials has permission for this call
  19. Check if the refund is not allocated.
  20. IsRefundAdminCharge should be false when refunding Season Delay Repay.

Example ProcessRefundApplication Request

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:ns="http://vixtechnology.com/cbe/2012-05-29">
   <soapenv:Header>
      <ns:UserCredentials>
         <ns:UserName>TSawyer</ns:UserName>
         <ns:Password>password</ns:Password>
      </ns:UserCredentials>
   </soapenv:Header>
   <soapenv:Body>
      <ns:ProcessRefundApplicationRequest>
         <ns:RefundNumber>17RYVKRCKP</ns:RefundNumber>
         <!--Optional:-->
         <ns:IsSmartCancellationRequested>false</ns:IsSmartCancellationRequested>
         <!--Optional:-->
         <ns:ReviewDecision>
            <ns:RefundDetails>
               <!--Zero or more repetitions:-->
               <ns:TicketRefundDetails>
                  <ns:TicketNumber>58</ns:TicketNumber>
                  <ns:TicketRefundStatus>Approved</ns:TicketRefundStatus>
                  <ns:ReasonCode>111</ns:ReasonCode>
                  <!--Optional:-->
                  <ns:SdciReasonCode>03</ns:SdciReasonCode>
                  <!--Optional:-->
                  <ns:RejectionReason xsi:nil="true" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"/>
               </ns:TicketRefundDetails>
            </ns:RefundDetails>
            <ns:IsRefundAdminCharge>true</ns:IsRefundAdminCharge>
         </ns:ReviewDecision>
         <!--Optional:-->
         <ns:AdditionalDetailList>
            <!--Zero or more repetitions:-->
            <ns:AdditionalDetail>
               <ns:Name>Requested by</ns:Name>
               <ns:Value>Registered phone number</ns:Value>
               <!--Optional:-->
               <ns:CreatedBy>TSawyer on behalf of MTwain</ns:CreatedBy>
            </ns:AdditionalDetail>
         </ns:AdditionalDetailList>
      </ns:ProcessRefundApplicationRequest>
   </soapenv:Body>
</soapenv:Envelope>
