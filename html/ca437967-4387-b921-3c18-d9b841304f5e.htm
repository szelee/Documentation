Remarks

BACKGROUND

This FEWS is used by Call Centre (CC) operators to find details of an existing account or confirm an email address does not already exist in the case of a new account.

The operator can search using a number of parameters and select an account for detailed review:

  • Customer name (Last name and first name).
  • Postcode.
  • Date of Birth.
  • Email Address.
  • Smartcard number.

PROCESS DESCRIPTION

This request from the Call Centre FEA to the CBE searches the customer accounts database and returns one or more summary account details to the FEA for the CC operator to identify the customer record required.

This request is used to verify that a customer account exists prior to the update and/or view of an account’s details. If the account is not found then this would trigger the creation of a new account via the front end application. (See Customer Account CC Creation – FEWS Ref 4.7).

The call triggers a search of the CBE customer database by means of one or more search criteria as shown below:

  • By name (last name, first name).
  • Postcode (of the customer’s primary address and not of delivery addresses).
  • Date of Birth.
  • Customer Email Address.
  • Smartcard number.

The call also supports searching by first name, surname or e-mail address substring. This means that if the characters provided are contained somewhere within the database field and the minimum configured number of characters has been entered, the customer record is returned. Where the minimum number of characters isn’t present on the customer account, exact matching is used.

The CBE outputs a list of customer account matches up to a limit of (n) where n is a configurable value to limit the volume of data that could be returned. The data is presented in a summary form for each matching account with customer name and address details, the security question and answer and optionally date of birth, email address and smartcard number where available. Where there are multiple accounts found, the Call Centre operator may use the security question and answer to uniquely identify the correct customer Where the configuration limit is exceeded with more record matches the CBE returns the number of records defined by the configuration limit.

The CBE will return a message when the search criteria used fails to find a match.

ASSUMPTIONS

  1. It is assumed that the FEA will ensure Customer Last Name, and ideally the Postcode, are the minimum search criteria to reduce the potential for large numbers of matches.
  2. It is assumed that where too many records are found to return to the FEA, the FEA should provide a suitable response requesting more search criteria, to improve the match in subsequent searches.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in appropriate error messages as shown in the appropriate API guide.

  1. Must have at least one search criteria present in input.
  2. Check the operator/user credentials has permission for this call Retail Channel must be Call Centre.
  3. If present: Smartcard number: (Smartcard number is checked using check digit validation.
  4. If wildcard searching is disabled in Config UI - Customer Email address (@ and . present).

Example Customer Account (CC) Search Request

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:ns="http://vixtechnology.com/cbe/2012-05-29">
   <soapenv:Header>
      <ns:UserCredentials>
         <ns:UserName>TSawyer</ns:UserName>
         <ns:Password>password</ns:Password>
      </ns:UserCredentials>
   </soapenv:Header>
   <soapenv:Body>
      <ns:CustomerAccountSearchRequest>
          <!--Optional:-->
          <ns:LastName>smith</ns:LastName>
          <!--Optional:-->
         <ns:Email>smith@google</ns:Email>
      </ns:CustomerAccountSearchRequest>
   </soapenv:Body>
</soapenv:Envelope>
