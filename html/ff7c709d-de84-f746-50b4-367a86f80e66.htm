Remarks

BACKGROUND

This FEWS allows the Operator to make an update to a specific refund record, which is in refund state ‘in progress’. This is required so that the operator can input or amend data into the refund record to:

  • Insert ticket numbers, which may not have been available when the refund record was created.
  • Amend ticket number.
  • Capture the data to identify the ticket for refund and also to get the data required for the SDCI BS record. This FEWS is required when the ticket has been issued by another TIS (TOD issue).
  • Allows the operator to update the review notes and optionally add a next review date.
  • Add details associated with Delay Repay of P2P or season tickets.

The FEWS contain provision for refund of foreign tickets (not sold by the CBE), as this is now required for the CZ channel.

This version of the FIGS supports the optional allocation of a refund to another operator to process.

PROCESS DESCRIPTION

This FEWS allows the Refund Operator to make an update to a specific refund record, which is in refund state ‘new’ or ‘in progress’ or ‘calculation in progress’. This is required so that the operator can input or amend data in the refund record to:

  • Insert ticket numbers, which may not have been available when the refund record was created, where the ticket number has not been available but has now been received by post.
  • Amend ticket number where previously incorrect.
  • Capture the data to identify the ticket for refund and also to get the data required for the SDCI BS record; this is required when the ticket for refund has been issued by another TIS (TOD issue).
  • Allows the refund operator to update the review notes and to add a next review date.
  • Add details associated with Delay repay of P2P or season tickets.

The refund record will be locked by the operator on requesting FEWS 9.1, it will remain locked in FEWS 9.1.2 and released and then locked again in FEWS 9.1. If an amendment of a ticket number has been done the operator must repeat FEWS 9.1.1. This approach ensures that only one operator can update a refund record at a time. If the request specifies that the refund case should be allocated to a different operator, the case should be moved to that operator and locked. The output then shows that Refund Allocated To field is returned where the CBE was requested to allocate a refund to another operator.

When the refund record is being updated the FEA must provide details of all tickets in refund application not just those for amendment. The output to this call is the refund number to signify the update has happened.

ASSUMPTIONS

  1. It is assumed that the refund operator has received the ticket(s) for the refund request from the customer and enters in one or more ticket details against the refund record.
  2. It is assumed that when an update is done all data items are resubmitted to the CBE otherwise previously entered fields will then empty.
  3. This FEWS captures foreign ticket data for refund where the original seller was not the CBE.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in an appropriate error messages.

  1. Operator (User credentials) exists and has refund administration rights.
  2. The Refund number supplied must exist.
  3. The refund must be in state of ‘new’ or ‘in progress’ or calculation in progress or calculation complete. There should be no change to status if the status is already CalculationInProgress or CalculationComplete.
  4. The ticket number must exist in sales transaction for refund.
  5. If TOD details are present then all TOD fields must be present.
  6. The Refund Category Type value should be one of the values in the list below.
  7. If the Refund Category Type value is season ticket refund delay repay then STDR outbound/inbound/both value must be present.
  8. If STDR value is present it should be one of outbound/inbound/both.
  9. Check the operator/user credentials has permission for this call.
  10. Check if record is allocated previously locked in FEWS 9.1.
  11. Check if sales transaction was TOD fulfilled when TOD details are present.
  12. The AllocateRefundTo field must be:
    1. For a valid CBE username.
    2. For an operator that has permission to do refunds.
    3. For an operator that is logged in.
    4. For a different user to the current user who has the refund locked.
  13. Where the fare is present as printed on physical ticket check the fare (in pence) matches the passenger fare data held in the refund record.
  14. Where the passenger status is present as printed on the ticket check that it matches the passenger status data in the refund record (created when the refund application was received).
  15. The foreign ticket detail must be absent where IsForeignFlag is TRUE (foreign ticket not sold by CBE). The retail channel is CZ.
