Remarks

BACKGROUND

This FEWS requests the details of a specific refund application from the CBE. It is typically expected to be used by a refund administrator to recall a refund record to review all the details of the refund and its processing status. This version has been updated to include details of any ETicket refunds.

BUSINESS PROCESS CONTEXT

PROCESS DIAGRAM

PROCESS DESCRIPTION

This FEWS supports the Refund Administrator with refund rights to retrieve a specific refund application for review. The refund may be new, or in progress, or to be processed. This is required so that the operator can view the refund record and decide what type of refund is required (in the next call FEWS 9.1.1).

The output from the CBE to the FEA contains all available information about the refund including the status of the refund, the reason for refund, the ticket details for refund, and the review notes recording the last refund update details.

The possible refund states for the refund application are:

  1. New
  2. In Progress
  3. Calculation Complete
  4. Approved
  5. Rejected
  6. Completed
  7. Refund Failed

Only refunds in these states are returned by this call. To avoid issues arising where a refund record is currently open and being processed the CBE uses record locking. This means that on request to the CBE the FEWS requests the record to be locked for editing and updating using a later FEWS 9.1.1. If the request does not request it to be locked then the record can only be read. If the record is already locked by another user then the refund record will not be available.

On receipt of this call the CBE checks the sales transaction record of the refund for the fulfilment type so that it can indicate in then output where a barcode ticket has been used. The CBE then sets the IsBarcodeTicket flag to true if that is the case otherwise it is set to false This data is then used by the operator to signify that they need to validate the electronic ticket using FEWS 9.1.4 before proceeding with the refund processing steps. The RefundAssessmentStatus data item will only be present if the ticket is an electronic ticket with barcode and a FEWS 9.1.4 validation check has already been done.

This call will be required also by ticket office staff to process a refund once the application has been created on the CBE database. (See assumption below).

ASSUMPTIONS

  1. It is assumed that the refund operator will use FEWS 4.3.1 Customer Sales Transaction Details (CC) to view the transaction, MOP details, fulfilment and journey details which relate to this refund application.
  2. This FEWS step may be hidden by the FEA for the ticket office so that the ticket office operator does not need to repeat the input details having just created a refund application (FEWS 9.0) for immediate refunds. The FEA will do this step behind the scenes without further operator input to retrieve the refund details for processing to the next step via FEWS 9.1.1.

VALIDATIONS

  • Parameter validation:
    1. All data items present
  • Business validation:
    1. Operator (User credentials) exists and has refund administration rights
    2. The Refund number supplied must exist
    3. The refund must be in state of ‘new’ or ‘in progress’.
    4. Check the operator/user credentials has permission for this call
    5. Check if refund has been allocated already.

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:ns="http://vixtechnology.com/cbe/2012-05-29">
   <soapenv:Header>
      <ns:UserCredentials>
         <ns:UserName>TSawyer</ns:UserName>
         <ns:Password>password</ns:Password>
      </ns:UserCredentials>
   </soapenv:Header>
   <soapenv:Body>
      <ns:GetRefundApplicationRequest>
         <ns:RefundNumber>17RFV09V6J</ns:RefundNumber>
         <ns:LockRefundFlag>true</ns:LockRefundFlag>
      </ns:GetRefundApplicationRequest>
   </soapenv:Body>
</soapenv:Envelope>
