ICbeTOServiceAmendReservation Method

This FEWS allows a passenger to amend an existing reservation either before the sale is completed or afterwards.

The existing reserved seats may be selectively changed by seat picks or by seat preferences. Advance tickets may only be changed by the TOC running the train and cannot be changed to a different train.

Flexible tickets may be changed to different seats on the same train, or to a different train on the same day.For changing to a different train it is assumed that the FEA will perform journey planning (FEWS 2.1) to check trains for which the chosen ticket type is valid.

Remarks

BACKGROUND

This FEWS applies to web and mobile channels only and allows the customer to change their reservation booking. This can either be done before the sale is completed or after the sale has been made. The customer can change their seat to another one of their choosing or leave it to the system to allocate another seat of the same type. The FEWS can also be used to delete a reservation if no change details are supplied.

BUSINESS PROCESS CONTEXT

This step can be done at any time before a sale has completed, after FEWS 8.1 make a reservation has been made, and a reservation booking reference returned. It can also be done after a sale has completed (after FEWS 6.2). The diagrams below show the various points and the sequence of calls for a number of scenarios.

PROCESS DESCRIPTION

This FEWS allows a passenger to amend an existing reservation either before the sales has been completed or after the sale. The key field required to support this is NRS booking reference. The customer can use this call to either delete an existing reservation by not providing any change data, or change a seat. For a journey with multiple legs and reservations, the input requires the customer to identify which leg (by origin and destination) is to be amended.

If they wish to change a seat to a chosen seat then the input requires the details of the new seat, if they want another seat to be chosen by NRS then this can be done without specifying new details.

If no amendment details are present then this request should be interpreted as a cancel reservation request. (This cancellation request can be done at the journey leg level rather than the whole reservation request as per FEWS 8.3.)

Other change data set is an optional group allowing any or all of the items in this group to be changed, if no data is entered then the existing data for these fields in the reservation record is unchanged in NRS.

If no new seat pick details are present just facilities change details then the CBE must request a new seat with these facilities and return the new seat details and NRS booking reference details. If no new seat pick details are present then any seat will be returned to replace the original ones.

If new legs, passengers or facilities are required then this FEWS is not used. These should be requested in a new reservation request.

If only one passenger out of a group of passengers requires to change their seat then this FEWS call will support this and other reservations relating to other passengers in the group will be unchanged.

If an amendment request to NRS fails, the original reservation on which the amendment was to be applied is retained with no changes to it. The CBE will provide a further error message in this case to indicate change of reservation not possible.

This document has been updated to include telesales retail channel as per VTEC customer requirements.

ASSUMPTIONS

  1. It is assumed that the FEA knows the list of accommodation and facilities codes that are required to be used in the input from FEWS 8.0 and 8.1 etc.
  2. It is assumed that the FEA has presented the customer with available seat options using FEWS 8.4 and 8.5 and the customer has identified a new seat as required where the journey is on VTEC services.

VALIDATIONS OVERVIEW

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields must be present.
  2. The reservation details (booking ref and journey booking reference) should exist.
  3. The date of reservation should not be in the past.
  4. Contact details if present should be email format.
  5. The sales transaction if present should exist.
  6. If the sales transaction number is present then the journey and leg details should exist within this sales record.
  7. The journey ticket type should not be an advance purchase ticket (cannot changequota controlled reservation).
  8. Either Amend booking details and/or Amend journeys should be present.
  9. If transaction number is present then check that the NRS booking reference relatesto this sale.
  10. Sales point details exist or User credentials are valid.
  11. Origin and destination locations cannot be an IPTIS group location.
  12. AssistanceTypeAttribute element is allowed for LNER Toc provider only.
  13. If AssistanceTypeAttribute is present:
    1. the maximum number of reservable companion seats in a single booking/journey must not exceed the maximum allowed value defined in Nrs part of ConfigUI (MaxWheelchairCompanionSeatReservations setting)
    2. the maximum number of reservable wheelchair spaces in a single booking/journey must not exceed the maximum allowed value defined in Nrs part of ConfigUI (MaxWheelchairSpaceReservations seting)
