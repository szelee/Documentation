Remarks

BACKGROUND

This web service call is requested by the Front End Application (FEA) for a Call Centre Operator via the Call Centre retail channel. The call requests large group fares for parties greater than 9 and less than 70 passengers, which are configurable values in the CBE configuration. It returns timetable and fares information (where available this will included discounted fares) for services from IPTIS. This document has been updated to reflect the CR100 changes to journey planning.

There is a requirement that the FEA is able to translate to and from CRS or NLC codes such that meaningful location names are presented to the user/operator. The necessary reference data is made available to the FEA by the CBE through separate reference data web service calls, see section 1.3.

BUSINESS PROCESS CONTEXT

PROCESS DESCRIPTION

The Call Centre operators working with group travel requests will use this web service request via the FEA to request fares and journey details for groups >9 and <70. Groups may be made up of children, adults or a mixture of both. The discounted fares returned by IPTIS may include reductions to one or both adult and child fares. The CBE input to IPTIS uses GRP as a railcard code to indicate group fares are required.

All group fares will be obtained from the IPTIS fares feed; these include child fares, The output fares provided may be reductions to fares as either a % discount or reduced flat fare, as set by the fare setting TOC and loaded into IPTIS. The CBE does not need to calculate any discounts. Not all journeys have group fares associated with them this depends on the fare setting TOC. If no group fare is available normal non discounted fares will be returned from IPTIS. The passenger status fields returned from IPTIS indicate whether a group fare discount is available, 060 for adult and 061 for a child indicates that a group discount has been applied. In addition the output includes the discount type to indicate that the discount is a group one. The output also includes a journey signature which is a feature of the IPTIS fares data. In cases where a train splits the RSID field returned in the output signifies this information.

If the input from the FEA is for passengers less than or equal to 9 passengers the CBE will still submit this request to IPTIS for journey and fares details and will return either small group fares or non discounted fares.

From a business perspective all group fares requests should be followed by a reservation request to NRS so that all passengers in the group have a seat reservation. The large group discount supported by IPTIS has the following properties: a. Applies to both adult and child faresb. Applies to standard class anytime & off-peak tickets onlyc. Fulfilment is by post onlyd. Excludes any railcard discounte. Applies on routes where the flow owner is the CBE Retail TOC i.e. (If CBE is a Virgin implementation, only include routes where Virgin is the flow owner.

When a valid Promotion Code is found and the Promotion Type equals “National”, the Promotion Code will be used in an IPTIS fares call as though it is a Railcard Code. IPTIS will return the correct passenger status codes and fares.

When a valid Promotion Code is found and the Promotion Type equals “Local”, the CBE will:

  • Return only the Ticket Types appropriate for the promotion as described by the promotions reference data.
  • Calculate the appropriate fare using the Discount Type and Discount Value in the Promotions Reference Table.

Fare Signature is a string that is returned within IPTIS Journey Planning response for each fare option. The fare signature contains the following information:

  • Ticket FTOT
  • Fare Setter
  • Route Code
  • Restriction Code
  • Fare Origin
  • Fare Destination
e.g. ‘SOS|IWC|00000|2C|0438|1072’

ASSUMPTIONS

  1. The Call Centre retail channel is assumed to be the group travel centre for the TOC. It is assumed that a small number of designated operators will be assigned to large group travel and will use this web service call.
  2. It is assumed that only standard anytime and off-peak fares qualify for Group discounts.
  3. Children in groups, get the agreed % discount as applied to the child fare, 16 and 17 year olds count as children on group fares, it is assumed that this is managed by the operator on input of number of passengers.
  4. Assumption, all group fares will be obtained via IPTIS fares feed, these include child fares and may be % discount or flat fares as set by the fare setting TOC and loaded into IPTIS. The CBE does not need to calculate any discounts.
  5. It is assumed that the trained group travel operator with suitable permissions will only sell group fares for the TOC that they represent and that the CBE does not control this.
  6. Non standard group fares in IPTIS are not supported by the CBE in line with other on line retailers.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. Retail channel is Call Centre.
  3. Station Locations are valid.
  4. All stations in input must be different locations.
  5. Outbound Date < n weeks from current date (n = booking window currently 13 weeks).
  6. Travel (Outbound) date+ time < Inbound date + time where there is a return leg.
  7. Ticket Class must be standard
  8. Number of adults and children is < n, where n is configurable and set to 70.
  9. The date of travel must be at least 7 days from the current date.
  10. Max No of Changes must be numeric of (0-9) where 9 means “Any” – if not provided the default value of 9 will be used.
  11. If AvoidStationLocation provided but type not specified – default value of “notvia” will be used.
  12. If EnforceRouteVia provided but type not specified – default value of “passing” will be used.
  13. If EnforceRouteVia type=”Route” then only one single 5 digit route code can be specified.
  14. If EnforceRouteVia type = changeat or callat then either (CRS and or NLC) must be present.
  15. Service Provider must be a valid two character service provider (as retuned in FEWS 10.5 Get Service Providers).
  16. If ShowOvertaken flag not provided, default to true.
  17. If ReduceLondonTransferTime Flag not provided, default to false.
  18. If InterchangeTimeIncrease provided value must be an integer between 1 and 120.
  19. InterchangeTimeIncrease and ReduceLondonTransferTime are mutually exclusive – i.e. both cannot be specified in the same input request.
  20. Check service provider is flow owner else reject this request.
  21. Travel Date must be less than or equal to the Extended Horizon End Date.
  22. Travel Date must not be an Excluded or Restricted date for the Extended Horizon.
  23. Origin and Destination must be a valid pair for the Extended Horizon.
  24. If Service Provider Filter is provided but type is not specified, default value is AtLeastOne.
