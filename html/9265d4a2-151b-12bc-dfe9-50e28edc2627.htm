Remarks

BACKGROUND

This web service call is required whenever a season ticket changeover is requested at a ticket office or via the Call Centre. A season ticket changeover event is done by the agreed business process of refund and rebook. This call covers the refund part of the process which is a refund of the unused part of the original ticket. The whole process is described below, which shows the rebook element as well.

BUSINESS PROCESS CONTEXT

PROCESS DIAGRAM

PROCESS DESCRIPTION

When a season ticket changeover is required – ie a change of journey or route or class etc - this call is used to calculate the refund element prior to a rebook of the new journey. This FEWS covers the refund part of the process which returns the refund value of the unused part of the original ticket. Unlike other refunds the calculation is based on the ‘base rate’. Depending on the Changeover refund response flag value, the output will either be the refund value (when the flag is set to True), or the refund value and refund reference number for the refund application created, (when the flag is set to False). This allows an operator to first present the refund value back to the customer prior to proceeding with the refund processing.

Only season tickets originally sold by the CBE will be available for the changeover process. Seven day season tickets are not applicable for changeover.

The period of validity (POV) of the season ticket cannot be changed by this process; the dates of the changeover tickets must be contiguous so there is no break between them. The original ticket must be refunded for the remaining time (POV) from the changeover date from. The new rebooked ticket which is requested using FIGS 5.7 will then run from the changeover date from (or next day if the request is done via ticket office.

As part of the input the operator must select a changeover types from the following list:

  1. a change of journey,
  2. change of route,
  3. change of class of travel,
  4. change of journey and route,
  5. change of journey and class,
  6. Change of class and route,
  7. Change of journey, route and class.

On receipt of this request (with the changeover flag set to FALSE, the CBE creates a refund record associated with the original sale. The record id will be required to complete the refund using FEWS 9.2 and then 9.4. The refund calculation is made using the fare in place at the time of the original sale.

ASSUMPTIONS
  1. It is assumed that the rebook of the changeover season ticket is done first using FEWS 5.7 and the refund is done using this FEWS followed by FEWS 6.2 for payment/refund, see the business context diagram.
  2. It is assumed that the FEA will make a single payment/refund call to the PSP via FEWS 6.2 for the net balance remaining (new fare - refund value) on approval of the refund and new sale.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. The transaction number must exist in the CBE database.
  3. Using the transaction number the associated ticket type must be a period season ticket or an annual.
  4. The remaining days must be >7 days (ie PoV – (startdate to current date)> 7 days)
  5. The changeover type is one of the agreed list above.
  6. If Changeover date from is provided confirm that Retail Channel is CC only.
  7. Check the operator/user credentials has permission for this call.
  8. Using the transaction number, check that the start date is in the past.

Copy
<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:ns="http://vixtechnology.com/cbe/2012-05-29">
  <soapenv:Header>
    <ns:UserCredentials>
      <ns:UserName>RegressionChangeover</ns:UserName>
      <ns:Password>password</ns:Password>
    </ns:UserCredentials>
  </soapenv:Header>
  <soapenv:Body>
    <ns:SeasonTicketChangeoverRefundRequest>
      <ns:TransactionNumber>17ABCD1234</ns:TransactionNumber>
      <ns:ChangeoverRefundResponseFlag>false</ns:ChangeoverRefundResponseFlag>
      <ns:ChangeoverType>Journey</ns:ChangeoverType>
      <!--Optional:-->
      <ns:ChangeoverDateFrom>2017-08-25T00:00:00</ns:ChangeoverDateFrom>
    </ns:SeasonTicketChangeoverRefundRequest>
  </soapenv:Body>
</soapenv:Envelope>
