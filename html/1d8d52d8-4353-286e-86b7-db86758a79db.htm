ICbeTvmServicePrintCtrTickets Method

The purpose of this FEWS is to enable a TVM, powered by the CBE, to request the printing of the ticket associated with a CTR record which has not been previously collected and printed. This FEWS includes the optional journey reference number so individual tickets for a journey within a CTR may be selected as required.

The FEWS also includes Fetched delivery print requests for a TVM following a walk up a sale as per SCCR058.

Remarks

BACKGROUND

The purpose of this FEWS is to be used for the collection and printing of CTR at a TVM where the TVM is powered by the CBE.

This FEWS includes the optional journey reference number within a CTR so one or more individual tickets for a journey may be selected as required.

Railway locations are defined using 3 character 'CRS' and 4 character 'NLC' codes, and the CBE uses these codes in the web service calls. There is a requirement that the FEA is able to translate to and from these codes such that meaningful location names are presented to the customer. The necessary reference data is made available to the FEA by the CBE through separate web service calls, see section 1.3.

This call has been modified in this version to support CCST direct printing at a TVM by means of ‘Fetched delivery’ (as required by SCCR058 and SCCR067 Fetched delivery part 2) as well as TOD collection. It also includes partial payment card PAN number to be checked when a CTR is being requested for TOD collection.

BUSINESS PROCESS CONTEXT

The business context for the various scenarios is shown in Fetched Delivery document.

PROCESS DIAGRAM

See CBE-00343 CBE Fetched Delivery Summary V4.3

PROCESS DESCRIPTION

This call covers a number of scenarios which are described below.

CTR Retrieval and Printing

This FEWS triggers the retrieval and printing of the whole CTR or selected journeys within the CTR, where the CTR has been previously placed on the LSM database by another TIS machine. This occurs when a ticket was purchased using TOD as the chosen fulfilment method and returns a CTR number to the customer to use when collecting the ticket from a TVM. The input includes the CTR or the CTR and journey reference together with the TVM sales location details.

It is required that the call FEWS 7.1.1 is made first by the TVM which then results in the CBE retrieving the CTR from the LSM and places a copy of the CTR record in a local cache. FEWS 7.2 then recalls the CTR from the CBE cache of CTR records, the CBE checks the status of CTR to ensure not already printed or collected (Booking/print flag should be 9/2 or P/2). The CTR record is not updated to show it has been printed until FEWS 7.5 call is received from the TVM by the CBE.

The CBE requests that the ticket is printed to the TVM printer at the designated sales point where the call originated, according to the agreed print specification CBE-00022. The output to the FEA signifies those journeys which have been sent for printing and the number of CCST coupons expected to be issued.

When the printing is successfully achieved by the TVM the FEA is expected to send a FEWS 7.5 to the CBE. On receipt of this call the CBE updates the CTR in the LSM database and releases the lock to show that all or just the selected journeys within the CTR have been ticketed. The CBE also creates the SDCI+ record for the TOD collection.

FETCHED DELIVERY PROCESS

This process shown below is only applicable to TVMs supported by the CBE, and shows TVM fetched delivery for walk up sales issued directly to CCST tickets.

ASSUMPTIONS

  1. It is assumed that this FEWS can be used for requesting a specific journey within a CTR or all journeys within a CTR, which are then fulfilled and printed.
  2. It is assumed that the TVM FEA first calls FEWS 7.1.1 for the CBE to retrieve the CTR and lock it and then write the record to the local cache.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. Only one of the following mutually exclusive parameters is allowed in request: Ctr Reference | transaction number.
  3. Where a CTR Reference has been input:
    • CTR record should exist in CBE cache.
    • The CTR is in the correct state for printing (ie has not been previously printed or collected).
    • One or more Journey Reference Number must be present and exists within the CTR reference.
    • The journey reference number if present within the CTR should not have been already printed.
    • All journey reference numbers associated with a CTR must be present.
    • The data in the CTR is valid.
  4. Where a transaction number is provided:
    • The sales transaction details should exist and the sales transaction state should not be cancelled.
    • The sales transaction should be in the correct state for printing – ie the print status should not be ‘complete’ or ‘complete receipt failure’ . The status should be either print pending or VOID (where it has previously been cancelled due to print failure).
