Remarks

BACKGROUND

This FEWS is the most complex and important FEWS in that it is the call which creates a sales transaction. It covers the customer’s choice of products in the shopping basket, passenger numbers, any discounts (railcards and promotions), together with their chosen fulfilment method. The possible products in a basket are as follows:

  • One or more Point to Point rail products which may contain:
    1. Multiple journey legs
    2. Associated reservations per leg
  • Optional associated Plusbus products
  • Optional associated Travelcards
  • Optional supplements/sundry items

Season ticket purchases are not included in this call but are covered in FEWS 5.1.

The fulfilment details provided in this request need to include the chosen the method of fulfilment together with any associated information in order for the CBE to successfully issue the ticket. For example where CCST (by post) is chosen the customer’s required delivery address is required. Where smartcard is chosen the customer supplies their smartcard number to be sent to the chosen delivery address (if they don’t have one).

On receipt of this input the CBE then validates all the data supplied and confirms the fares for each element and calculates total costs with any fulfilment charges, ready for the FEA to trigger the payment request to the PSP. The CBE also creates a sales transaction record in the database marked with status ‘awaiting authorisation. This status is updated once the payment is confirmed (see FEWS 6.2).

This call has been updated to include support for flexible ticketing of anytime or off peak tickets, which does not require as much input data as for other ticket types. Also added is support for one day travelcard sales sold without other rail products. Car parking is supported in this FEWS with a new structure to capture the required details.

This call has been updated as a result of CR100 which provides more information in the input regarding the detailed journey planning requirements of the journey being purchased.

The FEWS call can be made from all retail channels and is enhanced as Rev.3 as a result of CR114 .The call allows for allows the FEA to provide all the information required for fulfilment and SDCI settlement within this new Purchase Ticket request, thereby avoiding the need for CBE to obtain additional information by calling IPTIS. This is particularly relevant to the Onboard Mobile retail channel which does not depend on CBE/IPTIS to obtain its journey planning information.

The web service call has been renamed Purchaseticket3Request and includes the new structure documented in this integration guide.

PROCESS DESCRIPTION

This case covers the sale of point to point products and applies to all retail channels,. Prior to the sale being completed the CBE validates all the input data again including all items in the shopping basket:

  • The fare details for each journey and passenger type are correct
  • The fulfilment option chosen is feasible
  • The customer exists in the CBE database.

On successful validation the CBE creates the sale transaction record, puts the sale record into an awaiting authorisation state pending payment confirmation, (FEWS 6.2) and returns the sale record reference.

The basket of items may also include discounts which require checking, these may be one or more railcard type discounts including small groups (groupsave).

The input may also include sale of large group fares (greater than 9 passengers), these are described in FEWS Ref 2.3 and are not included in the examples this document.

If a reservation has been requested previously via FEWS 8.1 or 8.2; the Reservation Booking Reference is then provided in the input and is saved in the Sales Transaction details created as a result of this FEWS. The CBE Reservations Table entry created in FEWS process (FEWS 8.1 or 8.2), will already hold the Reservation Reference with a status of “Booked – awaiting confirmation”. This table is used to validate the Reservation Booking Reference if it is present in the input of this FEWS.

The required fulfilment details are provided in the input dependent on the retail channel. For web and call centre bookings where delivery of CCST tickets by post is requested, a delivery address number is supplied on input, with values 0 to n, for fulfilment method a, b, or c (see FEWS 6.0). These values refer to a corresponding address in the Customer Account details. 0 is the default value which indicates the Customer account address, but there are further delivery addresses supported. (The number of these is configurable but is currently 3). For efficiency of the On Board retail channel, the fulfilment option is either TOD or paper roll (CCST X type), and is determined by the On Board application and not by means of FEWS 6.0.

Where Smart indirect fulfilment has been selected in the input, if a Collection Location is not provided then the default location will be the Origin for the journey.

Where the input includes Group fare Flag = True, the discount is provided by a percentage or flat rate discount and are CBE configurable items.

This call has been updated to allow for more rapid flexible ticketing of anytime/off peak tickets, to reduce the input data required for these ticket types. For these tickets the journey information group need not be populated, as these are implicit in the Ticket Info group, see section 5.1.

If a Journey Signature is provided in the input then the CBE uses this to validate the journey call to IPTIS using an Exact Journey Planning Request (see section 5.12.1.1 of the IPTIS Interface Specification). In addition Travelcards (London underground) are now supported in this call; these may be either associated with a rail journey (inboundary ones), or may be standalone items and include non London Underground types.

The FEWS call also includes a new input structure to support car parking sales. The Station Description in this structure will be used on the Car Park Coupon as “Place of Validity”. This may be different to the Description associated with the NLC Code.

The output to this call is the transaction number, but in the case of the On Board retail channel it also includes the next SDCI transaction number to be used by the FEA when issuing the paper roll ticket. This is then confirmed back to the CBE when FEWS 6.2 is sent to the CBE so that SDCI transaction numbers are consistent and contiguous.

The only channel that can have ticket departure and arrival times in the past is On Board which may be due to offline delays. (Other channels may want this in the future but this is subject to further discussion regarding the proposed fraud checking facilities).

A further improvement in this version is to check that reservations match the journey legs they are required for, as there is the potential for the reservation in NRS to apply to a different train/journey leg from that which is being purchased The CBE now validates where a reservation reference is present in this call to check whether the RSID for each reservation within the reservation record has existing an journey leg. If not the CBE will throw an error related to Reservation RSIDs not matching with journey RSIDs.

Fare Signature is a string that is returned within IPTIS Journey Planning response for each fare option. The fare signature contains the following information:

  • Ticket FTOT
  • Fare Setter
  • Route Code
  • Restriction Code
  • Fare Origin
  • Fare Destination
e.g. ‘SOS|IWC|00000|2C|0438|1072’

This Rev.3 version of the call includes support for payment by evouchers, though the code does not yet support this (see example in section 6). It is envisaged it will be implemented shortly. The eVoucher’s remaining balances will be updated following this call once the purchase has been confirmed in FEWS 6.2. The eVoucher record at the journey level is captured for refund processing, fraud prevention, reconciliation and financial controls.

This Rev 3 version also includes support for sales with fulfilment by MTIcket and includes associated device details.

Only National Promotions (non railcards), split-ticket discounts or local promotions will be allowed. An individual passenger fare can only have a single DiscountDetails element which will either give details of a National promotion, or the most favourable discount available by either local promotion or split ticket. Group and Carnet discount types are not allowed.

If only a railcard discount is being applied (without local promotion or split ticket discount) then the Discount Type is Railcard, and the Discount Details element is populated as follows:

  • PassengerStatusCode will be appropriate for the relevant railcard.
  • Discount Id is not populated.
  • The DiscountedFare is not populated.
  • The HashedDiscountedFare is not populated.

Local Promotions will be applied with or without railcards depending on the discounting rule that will be configured in Admin UI by the TOC business. Details of the promotion discount will be provided in a DiscountDetails element associated with each Individual Passenger Fare. When the Discount Type is Local, the DiscountDetails element will contain the following:

  • PassengerStatusCode will always be 018(Adult) or 019(Child).
  • If a discount is applied without railcard, AdditionalDiscountRailcardCode will not be passed.
  • If a discount is applied with railcard, AdditionalDiscountRailcardCode will be passed and set to 3 character Railcard Code. This must be stored in the Railcard_Details table with reference to the Passenger Fares table so that it can be used to lookup Railcard Reference data and populate the Railcard status Code in SDCI BE record.
  • Discount Id is mandatory, as this will be used to obtain and validate any quotas have not been used up.
  • The DiscountedFare will be the fare the passenger actually pays, whereas the Fare in the parent Individual Passenger Fare will be either the original fare or the railcard discounted fare (if AdditionalDiscountRailcardCode is specified).
  • The HashedDiscountedFare will be a hash of the discounted fare that was provided in the GetFaresInformation response, which must be used to validate the DiscountedFare.

Split Ticket discounts will be applied with or without railcards. Details of the split ticket discount will be provided in a DiscountDetails element associated with each Individual Passenger Fare. When the Discount Type is SplitTicket, the DiscountDetails element will contain the following:

  • PassengerStatusCode will be appropriate for the relevant railcard if AdditionalDiscountRailcardCode is specified, else 000(Adult) or 001(Child).
  • If a discount is applied without railcard, AdditionalDiscountRailcardCode will not be passed.
  • If a discount is applied with railcard, AdditionalDiscountRailcardCode will be passed and set to 3 character Railcard Code. This must be stored in the Railcard_Details table with reference to the Passenger Fares table so that it can be used to lookup Railcard Reference data and populate the Railcard status Code in SDCI BE record.
  • Discount Id is not populated.
  • The DiscountedFare will be the fare the passenger actually pays based on the Split Ticket fare, whereas the Fare in the parent Individual Passenger Fare will be either the original fare or the railcard discounted fare (if AdditionalDiscountRailcardCode is specified).
  • The DiscountedFareHash will be a hash of the discounted fare that was provided in the GetFaresInformation response, which must be used to validate the DiscountedFare.

Where either a local promotion or a split ticket discount is defined, then validate that the DiscountedFareHash value can be recreated from the DiscountedFare value provided.

Where a split ticket discount is defined, the total basket fare is calculated as the total of the Fare values, and an eVoucher must be created which covers the difference between the summed Fare values and the summed DiscountedFare values for each basket item (so one EVoucher per basket item). These eVouchers have type SPLIT.

DepayRepayOptOut should be persisted in DB, with false as default value. Only transactions for which DepayRepayOptOut is false should be included in processing of automated delay repay.

DonateADRCompensation should be persisted in DB, with false as default value.

Where an optional BasketItemPairings list is defined, it defines pairs of single tickets in a basket to be linked as OutboundInboud transaction pairing types. Basic rules for basket pairing:

  • Multiple pairs can be linked - for example if a basket contains KGX - PBO, PBO - KGX, KGX - NCL, and NCL - KGX, these could be paired into two return journeys
  • Where a quantity greater than 1 is purchased, the basket items are paired together
  • Only single journeys where the origin and destination are exactly reversed can use this pairing
  • A pairing type 'OutboundInbound' is created for qualifying pairs
  • If a pairing is rejected, the reason for the rejection is included in the response. The Purchase Ticket request is halted in this case.

ASSUMPTIONS

  1. The choice of delivery address for postal fulfilment (CCST) will be supplied by the front end client using the delivery address flag, based on customer account address details.
  2. It is assumed that if fulfilment method is smartcard, the Front End application will be responsible for checking the smartcard used is in the Customer Account details or a new smartcard is requested.
  3. It is assumed that where TOD is chosen as the fulfilment option and the origin station does not have TVMs or “Collect at window” capability; the Front End application will manage the presentation of alternatives. A data set of TVM locations will be provided to the FEA web server at regular intervals by the CBE.
  4. It is assumed that the FEA will not support the selection of plusbus for the return leg of an open return ticket where the date is unspecified. (This is also trapped by business validation). Plusbus must always have a specified date associated with it).
  5. It is assumed that the FEA will validate date and time is in the future or prevent dates in the past to ensure that the customer is aware of this before purchase. (Not validated at the Web/Mobile FEA request to support purchases from outside the UK on different time zones). It is assumed that all FEAs will adopt this version (V23.0 and later) of the call as it allows faster and simpler data input for flexible walk up ticket sales.
  6. It is assumed that the FEA will use the skeleton customer account details FEWS 4.0.2 for rapid customer registration before FEWS 6.1 the sales process to create a partial account first before this call so that they can provide the new customer id as required in this call.
  7. It is assumed that the FEA will manage restricted days for Extended Horizon where there is an intention to release the ticket quota in the extended horizon but it has not been done yet. Customer may be asked if they would like to subscribe to be notified when quota is made available.
  8. It is assumed that the FEA will manage Excluded days for Extended Horizon where there is no intention to release the ticket quota for the extended horizon, eg. Bank Holidays.

VALIDATIONS

  1. Valid Date and time
  2. If present, smartcard number: (Using check digit validation (See ITSO TS 1000-2:2010-02 on ITSO website.)
  3. All mandatory items must be present
  4. Charity Id length must not exceed 36 characters
  5. Customer Reference Number exists
  6. Origin and Destination Location codes are valid
  7. Validate fares, but not for validate fares for OnBoard channel
  8. Fulfilment method must be a valid fulfilment method (ie in the fulfilment table)
  9. If Fulfilment method flag is CCST (any type), then Delivery Address Number must be present.
  10. If Fulfilment method flag is smart (any type) validate collection location is smart enabled.
  11. If supplement details are present, then fulfilment option can only be TOD or CCST (any).
  12. Check total number of passengers not greater than that configured for the Retail Channel, currently:
    1. Web = 1-9
    2. Mobile = 1-9
    3. Call Centre = 1-69
    4. TVM = 1-9
    5. Ticket Office = 1–69
    6. On Board = 1-9
  13. Validate Total Basket Value
  14. If Retail Channel is Call Centre or ticket office or on board then Operator ID must be valid and logged on with the correct access role. If retail channel is web/mobile or TVM then sales point details must be present. Check that only one of user credentials and sales point details is present.
  15. If fulfilment type is self-print or mobile (g) then Passenger name must be present
  16. Where Reservation Booking Reference exists, check the reference exists in the CBE Reservations Table. The OnBoard channel does not allow for reservations. Reservation Booking Reference for the channel thus has to be empty.
  17. Reservation booking reference field must be unique and associated with a journey in the basket.
  18. Where the Group fare is used, the number of passengers must be >9 and <70
  19. Where the Group fare is used, the Fare Category must be Standard or Off-Peak
  20. Where a promotion code has been input, using the CBE Promotions Reference Table (See FEWS 10.3)
    • The Promotion Code must exist in the Promotions Reference Table
    • The current date of sale is within the associated selling window dates (from and to) for the promotion.
    • The journey dates outbound and return are valid for the effective “From” and “To” the associated journey dates for the promotion in the table.
  21. Where supplement items are included, check that the item is valid for the journey exists in IPTIS.
  22. Where ticket type is open return (no return date specified), plusbus can only be present for each origin and destination location for outbound journey date.
  23. Check the operator/user credentials has permission for this call (see latest version of FEWS Permissions)
  24. Carnet flag if set to true, number of passenger should be one of a configurable list (currently 5, 10 values)
  25. Transaction DateTime should be empty for all but OnBoard retail channel. It is allowed to be in the past. Empty value means “now” for the CBE.
  26. OnBoard channel specific validations:
    1. PaperRoll and Tod are the only allowed fulfilment methods
    2. No reservation booking is allowed regardless of fulfilment method
    3. Additional items are supported only for ToD fulfilment
    4. Group fares are allowed
    5. Other charges are not allowed; the reason is creation of BP sdci record which may break contiguous ticket numbers
  27. Apart from the Onboard retail channel the journey departure time/arrival time should not be in the past.
  28. The Journey Information must be present for advance purchase tickets.
  29. For Car Parking Tickets the Ticket Type will be “ZVG”
  30. For Bike reservations the Ticket Type will be “ZVC”
  31. For Parking Details, Vehicle Registration number is mandatory if Ticket Type is Car Parking (ZVG) and is a Season Ticket (Validity > 1 day).
  32. Where a reservation reference is included check that the reservation details match the journeys being purchased. (See note 17 below)
  33. The lead time between travel date and current date is sufficient for posting (see fulfilment table)
  34. Validate the railcard code against the customer account and if the same then check the expiry date of the railcard is greater than the date of outbound travel.
  35. When entered, the eVoucher Code must exist and be associated with the customer ID in the Request and have a status of “Claimed”.
  36. In case of M-Ticket fulfilment, the CBE should validate the device id passed in the request is valid and linked to the signed-in customer’s account.
  37. If collection location is present AND smartcard is the chosen the fulfilment method then validate that the location is smart enabled as per RCS, otherwise ignore this field as not required for other fulfilment methods.
  38. The eVoucher must not be expired or Voided.
  39. The following 3 additional business validation checks only apply to the web retail channel for VTEC, where the CBE local table of extended horizon dates is available, otherwise should be ignored.
    • Travel Date must be less than or equal to the Extended Horizon End Date.
    • Travel Date must not be an Excluded or Restricted date for the Extended Horizon.
    • Origin and Destination must be a valid pair for the Extended Horizon.
  40. If Sales offering context is not empty, then the PRM_ENABLE_SALES_OFFERING (Config UI) must be set to TRUE.
  41. If Sales offering context is not empty, then the sales offering Code is valid and current.
  42. If Access control id is not empty, then Access control id must be valid.
  43. If there is any Local Promotion discount in the request, then the Sales offering context cannot be empty.
  44. If there is any Local Promotion discount in the request, then all Discount ids must be valid.
  45. If PassengerGroupFares is present this is validated as follows
    1. Ticket Type is a group ticket type on the RJIS ticket types file (MAX_PASSENGERS > 1)
    2. Group Ticket Adults is valid for the ticket type (>= MIN_ADULTS & <= MAX_ADULTS)
    3. Group Ticket Children is valid for the ticket type (>= MIN_CHILDREN & <= MAX_CHILDREN)
    4. The number of passengers is valid for the ticket type (>=MIN_PASSENGERS & <= MAX_PASSENGERS)
    5. Note: there may be > 1 group fare e.g. 2 DUO tickets (4 adults in total) or 2 Chiltern Family Tickets (for instance 4 adults and 3 children) – each group fare should be validated as above
  46. If Group Railcard discount is specified, the number of passengers must be within the MIN & MAX passenger parameters for the railcard supplied.
  47. If there is any Local Promotion discount in the request, then the Sales offering quota cannot be exceeded
  48. If there is any Local Promotion discount in the request, then the Sales offering customer quota cannot be exceeded
  49. If there is any Local Promotion discount in the request, then the Discount ticket quota cannot be exceeded
  50. If there is any Local Promotion discount in the request, then the Discount customer ticket quota cannot be exceeded
  51. TicketUtn must be provided for OnBoard mobile retail channel for all transactions that are fulfilled to PRT. Ticket Utn must be a 13-character string accepting only alphanumeric characters and spaces.
  52. If no journey details are specified (flexible ticket) then Uts Zone, Category and Fare Source are mandatory.
  53. When optional BasketItemPairings array is included, check each BasketItemPairing item for the following:
    1. BasketItems with ItemNumbers specified in OutboundItemNumber and InboundItemNumber of BasketItemPairing must exist
    2. BaskeITems paired by OutboundItemNumber and InboundItemNumber have reversibly same origin and destination stations defined
    3. No duplicates are allowed in BasketItemPairings array (it means, that all OutboundItemNumbers and InboundItemNumbers must be unique)
