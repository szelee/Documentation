Remarks

BACKGROUND

This FEWS allows the user to update their account details held in the CBE database. All data previously captured in FEWS 4.0 customer account creation can potentially be updated as required and resubmitted to the CBE to update the customer record.

The current customer account information, held by the CBE, is supplied to the FEA on receipt of a valid customer login request (FEWS 4.4) as shown below).

BUSINESS PROCESS CONTEXT

The diagram below shows the context of where in the Customer Account Update process this FEWS call occurs.

PROCESS DIAGRAM

PROCESS DESCRIPTION

This FEWS allows the customer to update their account details that are held in the CBE database. This web service call is made following a successful customer logon web service call; the response to the successful customer logon supplies the full customer record to the FEA. The FEA will control the ability of the customer to update the record details as required. All fields shown in the input, excluding the password, can be updated using this web service. Password updates are handled by separate web service calls. The CBE will update all fields supplied by the FEA in this web service call. Where a previously supplied field is returned empty, the customer record will be updated with that data element empty, so therefore it is important that the FEA returns all information and not just the data elements that have been updated.

The output from the CBE is the full set of customer account attributes following the update and the customer id as used by the CBE. It also includes 3 additional fields which are not available for update by this FEWS namely:

  • Tokenised Payment Card Count
  • Customer account type (full/guest)
  • Customer account creation date

ASSUMPTIONS

  1. It is assumed that the Front end will perform PAF validation of the postal address – this is not the responsibility of the CBE.
  2. The Front end application will be responsible for managing or limiting the changes to the customer account details and for any security checks prior to submitting the account details to the CBE.
  3. It is the responsibility of the FEA to enforce the default value of the direct service flag to ‘all services’ as input to the CBE, if the customer does not respond to this field as this is mandatory.
  4. Should the password field element be shown in the request this must be filtered out by the FEA.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. Transform email address into lower case, check it is unique and store against customer account.
  3. If Ticket Class is present values should be should be First, Standard (nulls=All).

  4. Customer Address validation:

    1. Must consist of Postcode and Address Line 1 and 2 for all countries
    2. Postcode should be 4 to 10 alpha numeric characters
    3. For USA addresses line 4 (City), Line 5 (State) and Postcode (ZIP) must not be blank.
    4. Address type must be one of billing/delivery/contact
    5. Addressee name should be present if address type is delivery
  5. Age of customer must be => than 16 years.
  6. Seating preferences if present exist

  7. Check Railcard:

    1. if present it exists see FEWS 10.6
  8. Ticket type exists.
  9. If Loyalty card details group is present then all fields within must be present.
  10. If present Check Loyalty scheme name is a valid one (Nectar and Virgin Atlantic Flying Club)
  11. Number of loyalty cards being added should be within max limit configured against the scheme (to be advised).
  12. If more than one loyalty detail is present then CustomerLoyaltyCardsId must be unique.
  13. Check Phone number is present if contact preferences field states Phone (not mobile)
  14. Check security details if present do not exceed 3
  15. Journey name should be unique to the customer account
  16. If Journey type (in favourite journeys) is Season then outbound and return details should be absent.
  17. If Journey type (in favourite journeys) is P2P then outbound details must be present.
  18. Customer loyaltycard detail was not found by identifier or it does not belong to customer.
  19. If IsLoyaltyOptIn is true then any loyalty evoucher types can’t be claimed
