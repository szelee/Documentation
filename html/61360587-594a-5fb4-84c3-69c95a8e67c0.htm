Remarks

BACKGROUND

This FEWS adds or updates tokenised payment card details against an existing customer account in the CBE database. The call applies only to the web and mobile retail channels where Card Holder Not Present (CHNP) payment card transactions arise.

As part of CR099 this FEWS has been updated to allow multiple payment cards to be added and associated details provided. The output now returns all card details set up.

BUSINESS PROCESS CONTEXT

This call can be made at any time. Diagram not required.

PROCESS DIAGRAM

Not required.

PROCESS DESCRIPTION

This call adds or updates any tokenised payment card details in an existing Customer Account in the CBE database. It then allows the token to be used for card payments in FEWS 6.2 for sales of rail and associated products, instead of full payment card details. The input includes the billing address associated with the payment card where a new card token is being supplied.

At least one of card details or billing address details must be present, either to be added or to be amended. If only card details are present then these must include an address id for the card to be associated with for billing. Where address details are provided in the input they are then associated with the customer account if not already present (e.g. for skeleton accounts) and marked as used for card billing. If an existing address (address id provided), is input, which is to be used for billing then the CBE adds this to the addresses held with type billing in addition to any existing address types already set up. If a new address is added in the input and the IsDefaultBilling flag is set to True then the CBE checks other addresses (billing) where this flag has been set to True and amends it to False.

Where a new card detail input has default card flag set to Y/TRUE then the CBE checks other cards and amends the default flag against other card to N/FALSE and marks the new one as the default card in the customer’s card details. If the value is N/FALSE then the CBE does not update the default flag against the chosen card. If the token passed on the request to the CBE matches the token currently stored against the card id, then only the Card Expiry Date and Address Id can be updated with the values in the request.

The output from the CBE is the customer id and the new customer account card and associated billing address details to signify a successful update to the customer account.

ASSUMPTIONS

  1. It is assumed that the FEA will manage the input of partial PAN details and will support the tokenisation process with the PSP.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in appropriate error messages as shown in the appropriate API guide.

  1. All mandatory fields are present.
  2. Customer Id exists (and may be partial/skeleton or guest or full customer account).
  3. If non-zero Address Id is specified check:
    • Address Id exists
    • Address Id belongs to the account specified by Customer Id
    • Address has the address type value Billing in customer account address details
  4. If Billing address is passed:
    • Must consist of Postcode and Address Line 1 and 2 for all countries
    • Postcode should be 4 to 10 alpha numeric characters
    • For USA addresses line 4 (City), Line 5 (State) and Postcode (ZIP) must not be blank
    • Address type must be billing
  5. If non-zero Card Id is specified:
    • Card Id exists
    • Card Id belongs to the account specified by Customer Id
  6. Either card details and /or billing address details must be present both cannot be absent
  7. If card details are present but address details are absent then billing address id must be present in the card details
