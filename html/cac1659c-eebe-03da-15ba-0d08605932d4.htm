Remarks

BACKGROUND

The purpose of this FEWS is to enable a customer at a TVM to view the journeys in a previously placed CTR. It also includes a check of the payment card used in the LSM database against that supplied in the input.

PARAMETER VALIDATION

Truncated PAN format

BUSINESS VALIDATION

  1. All mandatory fields are present.
  2. The sales point details are valid within the CBE set up. (Combination of NLC, window number, machine number)
  3. CTR number should exist in LSM database.
  4. If the card details input do not match the partial PAN details in the CTR AND the mailing flag is set to 1 then reject this call.
  5. If the card details input do not match the partial PAN details in the CTR AND the mailing flag is set to 3 then do NOT reject this call.
  6. If the CTR record in the LSM database does not contain card details then truncated PAN details do not need to be present nor checked.
  7. If the CTR record in the LSM shows more than one payment card is used then multiple truncated PAN details must be provided.
  8. The CTR record should not already be present in the CBE cache.

NOTES

  1. The CBE should output details of all journeys within the CTR regardless of their status
  2. In the case where the LSM database is not available the CBE receives an Internet connection failure:
    XML
    Copy
    <ErrorCodeMessages>
<ErrorCodeMessage>
<ErrorCode>903</ErrorCode>
<ErrorMessage>Ctr internal error</ErrorMessage>
<Tag i:nil=”true”/>
</ErrorCodeMessage>
</ErrorCodeMessages>
    When the an incorrect url is used to access the CTR the error message is
    XML
    Copy
    <ErrorCodeMessages>
<ErrorCodeMessage>
<ErrorCode>908</ErrorCode>
<ErrorMessage>Connection to third party interface (Ctr) failed.</ErrorMessage>
<Tag i:nil=”true”/>
</ErrorCodeMessage>
</ErrorCodeMessages>
    In both these cases these responses to the CBE then need to be translated by the CBE to the FEA as shown in the output to the FEA above CTR access issue
  3. It has been agreed with SC and PKN that the CBE will manage the checking of the payment card if the mailing flag is set to 1 (Same card needed for issue, from self- service machine or staffed collection point, as used for purchase). This is as per RSPS5040 version 3 message set using partial PAN (the first 6 and last 4 digits of the PAN).
  4. On receipt of a valid CTR and partial PAN the CBE should check the CTR mailing flag –
    A. if = 3 then create the print message and return it as output and do not check against the truncated partial PAN.
    B. if = 1 then check the partial PAN input matches the J5-CC5- CardNum details then create the print message and return it as output OR if not then output error message ‘card does not match payment card’.
  5. If the CTR has previously been recalled and issued then the CBECTRStateDetails will be present, this provides additional information regarding state changes during the printing process will assist in cases where issues with collect have arisen. This data is maintained by the CBE and supplements the LSM CTR data.
  6. CTRStatus

    On receipt of this request the CBE should retrieve the CTR from the LSM database and write it to the CTR cache within the CBE database. if the CTR has not been fully printed LSM will automatically lock the CTR. However if the CTR has already been fully printed then LSM will not lock the CTR , i.e. this is a recall of the CTR post printing.

    If CTR cannot be recalled from LSM because it is locked
    CTRStatus = “LockedByOther”
    Else
    If CTRPrintStatus = 3 and CTRBookingStatus = 4
    CTRStatus = “UnLocked”
    Else
    CTRStatus = ”Locked”
    (see RSPS5040 section 2.8)

  7. The mailing flag check of the CTR can be done first by the CBE and if it is set to 3 any payment card then the card details do not need to be checked against the Partial PAN held on the CTR. If however the mailing flag is set to 1 then card details must be checked that they match.

ASSUMPTIONS

  1. It is assumed that in the event of CTR access issue message arising the FEA will generate a suitable error message and allow the user to continue within the application. (This is an RSP accreditation requirement).
  2. It is assumed that the FEA will display the ticket types name from the output FTOT details by using FEWS 10.8 Get Ticket Types.
  3. It is assumed that the TVM FEA will manage whether a truncated payment card is input to the CBE, it is shown as optional above but the FEA may wish to make this a mandatory feature of the TVM. The CBE logic above supports different and multiple MOPS used at time of purchase hence the optional structure in the input.
  4. It is assumed that the FEA (TVM) will ignore details of journeys already printed.
