ICbeTOServiceDoNonissue Method

This FEWS sends a non-issue message to the CBE in case of a problem issuing CCST ticket at the Ticket Office, Call Centre, or On Board, so that the ticket coupons can subsequently be cancelled and the transaction reversed in Lennon (by a non-issue record). The non issue value is used to indicate the type of problem with the ticket being resolved.

Remarks

BACKGROUND

This call by the FEA to the CBE informs the CBE that a non-issue type event has occurred. A non-issue is a term coined by RSP to describe a reversal of ticket sale, typically where the ticket has not been able to be fulfilled due to printing issues or the wrong product has been issued and needs to be reversed. This call to the CBE in turn triggers various actions by the CBE to effectively reverse a ticket sale which typically failed to be fulfilled properly.

BUSINESS PROCESS CONTEXT

This FEWS occurs after a ticket has been sold i.e. after FEWS 6.2 has completed successfully. The diagram below shows the typical steps in a for the ticket office retail channel non-issue where any part issued ticket is cancelled and the ticket is then reissued. The sequence of events for other retail channels will differ.

For the On Board channels the non-issue call FEWS 9.7 will be followed by FEWS 9.7.3 with intermediate steps missing as there is no action for the CBE to do to support overprinting. (See section 3.2 for more information).

PROCESS DIAGRAM

Not required

PROCESS DESCRIPTION

This FEWS sends a non-issue request message to the CBE in case of a problem issuing CCST ticket at the Ticket Office or Call Centre. The same call is used by the On Board channels when the printing has failed or then the wrong product has been issued., . The non-issue reason code is used to indicate the type of failure in printing the ticket. The codes used are Paper Jam, Print Failure, Wrong Product, Disruption, Other. The input includes the original sale transaction number or in the case of On Board channel, the ticket number which has failed is provided.

In the ticket office and call centre this call is followed by a check of the mag stripe on any part issued tickets (FEWS 9.7.1), then an overprint request is sent to the CBE and then onto the print server (FEWS 9.7.2) to cancel the CCST ticket(s) which have been partially printed and cancels the magnetic stripe. FEWS 9.7.3 then confirms that that the overprint has completed and triggers the SDCI creation in Lennon creating either a BN or a BF record as appropriate.

As agreed with RSP this approach will be used for ticket issuing problems, rather than as means of a refund which is covered by other processes. The non-issue process reverses the previous sale and typically this then is followed by a reissue request when the printer issue is resolved (FEWS 9.7.4)

The output from the CBE on receipt of this call is just an ACK except where the retail channel is On Board when the SDCI record count is returned with the sale transaction number (of the original sale). This then allows for the On-board channel to be off line, in that the original sales transaction number may not be available to the device, but the ticket number which has failed will be known, as this is managed by the device.

ASSUMPTIONS

  1. It is assumed that in the case of a walk up sale at a TVM which fails, the TVM will manage this by FEWS 7.6 and refund the customer so that the customer does not need to go to the ticket office.
  2. As per RSP guidance it is assumed that the ticket operator only uses the non-issue process for ticket issue failures and not for refunds, except where the reason for refund is for exceptional circumstances as defined by local instructions, (e.g. Service disruptions), or where an in-shift refund may be required.
  3. It is assumed that a trained operator knows that all tickets in a transaction have to be non-issued and reissued, as per current Lennon operation for non-issue and reissue.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Failure to meet these checks will result in appropriate error messages as shown in the appropriate API guide.

  1. User credentials check, and present for retail channel TO/CC/OnBoard.
  2. Non-issue reason code valid.
  3. Transaction Number exists in CBE.
  4. Ticket Number must be present if RC=OnBoard.
