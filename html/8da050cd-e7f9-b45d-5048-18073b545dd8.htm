Remarks

BACKGROUND

This FEWS supports the operator with suitable permissions for refund to retrieve the list of refunds from the database filtered by input parameters to get the associated detailed information in order to review the refunds (FEWS 9.2). This FIGS has been updated to include additional payment details where they are present to reflect card and voucher repayments.

This version has been updated to include details of repayment by evouchers. It also includes details of the allocated operator where a refund record is already being worked on.

BUSINESS PROCESS CONTEXT

PROCESS DIAGRAM

Not required.

PROCESS DESCRIPTION

This FEWS supports the Refund Administrator with refund rights to retrieve the list of refunds from the database filtered by input parameters to get the associated detailed information, in order to review the refunds (FEWS 9.2). It allows the Refund Administrator to retrieve refunds by a designated Refund search criteria:

  1. refund state
  2. refund Number
  3. operator
  4. customer.

It also allows the Refund Administrator to select the page size and number of items on each page, to manage the potentially large output. The CBE retrieves the refund request details according to the search criteria requested. The output is returned as an ordered list by refund Id (which is sequential by date). It is not possible to request all refunds, and therefore a refund status must be set in order to limit the volume of data returned.

The operator can view failed refunds (where payment has failed), to progress them by resubmission of FEWS 9.4. The review date and the review notes will store the failure details from FEWS 9.4 where a failure has occurred.

The ticket type provided in the output is an FTOT (e.g. SOS standard open single), however for rangers an rovers there is no FTOT just a code which is provided as an alternative only in the case of these products.

This FIGS includes updated payment details where the refund has been processed and refunded, in the case of card repayments and vouchers. The input now includes a new Boolean field IsAllocated which is used to either include those refunds that are already allocated (locked) to another operator (i.e. a different operator to the one in User Credentials). The FEA may then choose to grey-out those refunds currently being worked-on by another operator. 0 means exclude refunds currently allocated (locked) to another operator. The allocated operator is the operator currently allocated to this refund.

ASSUMPTIONS

  1. It is assumed that FEA will manage the display of the refund details returned by the CBE and the history of refunds only displaying those most recent as required.
  2. It is assumed that refund agent name details etc in the refund agent pair data is not to be displayed or presented by the FEA where the query is from a customer via the web. Other details are shown to include lower levels of granularity with the refund processingWhere more details of an individual refund is required it is assumed that FEWS 9.1 will be used.

VALIDATIONS

  • Parameter validation:
    1. None
  • Business validation:
    1. Operator (User credentials) exists and has refund administration rights
    2. One refund search criteria is present
    3. Page number and page size cannot be less than 1
    4. All mandatory fields present
    5. If Callcentre Retail Channel then Check the operator/user credentials has permission for this call (see latest version of FEWS Permissions)
    6. If Web/Mobile retail channel then check sales point is valid
    7. If Web/Mobile retail channel, the only search criteria permitted is Customer Id and Customer Id with an optional Refund Number
      1. Customer Id must be present and exist
      2. Refund Number (optional) must belong to the Customer Id passed in the request
    8. Check the operator/user credentials has permission for this call (see latest version of FEWS Permissions)

<soapenv:Envelope xmlns:soapenv="http://schemas.xmlsoap.org/soap/envelope/" xmlns:ns="http://vixtechnology.com/cbe/2012-05-29">
   <soapenv:Header>
      <ns:UserCredentials>
         <ns:UserName>TSawyer</ns:UserName>

   <soapenv:Body>
      <ns:GetRefundListRequest>
         <ns:IncludeAllocated>true</ns:IncludeAllocated>
         <!--Optional:-->
         <ns:RefundNumbers>
            <!--Zero or more repetitions:-->
            <ns:RefundNumber>17RSPZUV4P</ns:RefundNumber>
         </ns:RefundNumbers>
         <!--Optional:-->
         <ns:RefundStatuses>
            <!--Zero or more repetitions:-->
            <ns:CbeRefundStatus>Completed</ns:CbeRefundStatus>
         </ns:RefundStatuses>
         <!--Optional:-->
         <ns:RefundSubStatuses><!--Zero or more repetitions:-->
            <ns:RefundSubStatus>Done</ns:RefundSubStatus>
         </ns:RefundSubStatuses>
         <!--Optional:-->

         <!--Optional:-->
         <ns:Operators xsi:nil="true" xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"/><ns:Customers><!--Zero or more repetitions:-->
            <ns:CustomerId>1190</ns:CustomerId>
         </ns:Customers>

</soapenv:Envelope>
