Remarks

BACKGROUND

This FEWS is used to get either the fare for the new journey when a changeover of a season ticket is required, or to purchase the new ticket for the new journey as part of a changeover process. It is done at a ticket office or via the CC. The fare for the new ticket is calculated using the base rate as described in FEWS 9.9. On completion of this FEWS the purchase of the ticket can be made with FEWS 6.2.

The changeover is done by means of rebook (new journey) and refund (the old journey by means of FEWS 9.9.

The request may be for a changeover for a change of journey, route or class of travel or combination of these. This also includes changes from a Season Ticket to a Travelcard or vice-versa.

PROCESS DESCRIPTION

The request may be for a changeover for one of the following:

  1. Change of journey

  2. Change of route

  3. Change of class of travel

  4. Change of journey and route

  5. Change of journey and class

  6. Change of class and route

  7. Change of journey, route and class

Only tickets originally sold by the CBE will be available for changeover, this is a limitation due to the fact that the CBE cannot refund ‘foreign tickets’ – those not originally sold by the CBE which is the 2nd part of the changeover process (rebook and refund).

The ticket type must be a period season ticket or an annual, 7 day ones are not permitted to be changed. The dates of the changeover tickets must be contiguous so there is no break between them. The original ticket must be refunded for the remaining time (POV) from the changeover date from (see FEWS 9.9). The new ticket will then run from the changeover date from (or next day if the request is done via ticket office).

In a ticket office the changeover date is assumed to be from the next day of the request, for a CC the changeover from date must be supplied.

This FEWS supports 2 options for either a fare enquiry for the new journey, or the fare and the creation of new sales record, the output depending on the changeover response flag (Boolean. if the flag is TRUE then only the new changeover fare is output, if FALSE then both fare and sales record reference is output.

The PhotocardId is not required to be validated against the customer account. if it is provided it will update customer's PhotocardId in database, if it's not provided it will use value from database. The PhotocardId is optional as it is not required for the first call when the Change over Response Flag is set to TRUE and only a fare is requested. The PhotocardId is needed in second call of 5.7 when the Changeover Response Flag is set to FALSE.

ASSUMPTIONS

  1. If the request is made by the CC then the FEA or operator must ensure that the dates for the changeover are sufficiently in the future to allow the tickets to be returned. The CBE will not manage this.
  2. It is assumed that changeover of duplicate season tickets is managed by a suitable business process which results in a monetary refund only and that changeover of these with the issue of a new ticket is not supported. This is in line with the options in the retail manual.
  3. This call assumes that where the changeover response flag is set to FALSE the FEA will have previously performed FEWS 6.0 to get fulfilment options for the changeover journey. Where the response flag is TRUE (fare only enquiry) then this step is not required.
  4. This FEWS assumes that when an initial changeover fare enquiry is first requested the next step to progress to a sale is FEWS 6.0 for the available fulfilment options. Following this the FEWS 5.7 is repeated including with chosen fulfilment details provided and the response flag set to FALSE. The sale is then completed with the payment/purchase confirmation FEWS 6.2. This approach should simplify the design of the FEA for changeover requests.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with appropriate error message(s) as shown in the appropriate API guide.

  1. All mandatory data items are present.
  2. The ticket number must exist in the CBE database.
  3. The ticket type must be a period season ticket or an annual.
  4. The remaining days must be >7 days (ie PoV – (startdate to current date)> 7 days)
  5. The changeover type is one of the list below.
  6. Origin and destination NLC/CRS validation as previous input FEWS (i.e. Journey Planning), NLC must be used for ticket origin and destination location.
  7. Check the route code exists for the new journey.
  8. Check Ticket class exists.
  9. If Changeover date from is provided confirm that Retail Channel is CC only.
  10. If the changeover response flag is FALSE (ie fare and sale response) then the fulfilment details must be present.
  11. Check Fulfilment details (where present) for journey route in RCS.
  12. Check the operator/user credentials has permission for this call (see latest version of FEWS Permissions).
  13. If Changeover Response flag = FALSE the Changeover Refund reference exists in the CBE database
  14. If Changeover Response flag = FALSE the Ticket number must match ticket number in the refund record.
  15. The start date must be in the past.
  16. Where PlusBus details are included:
    • The Period of Validity should only be; 7day, 1 month, 3 month or annual.
    • The Journey and PlusBus ticket validity must match the season ticket duration.
