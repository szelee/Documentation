Remarks

BACKGROUND

This FEWS allows any FEA to enquire whether a journey is possible at the times requested by the user, and provides details of the train times and route information in response to the request.

Journey Planning requests use three character codes (also known as CRS codes) to identify the origin and destination locations, along with identifying any points along a possible route that a passenger may wish to travel via or to avoid. Examples of CRS codes are, ‘CRE’ for Crewe, ‘PAD’ for London Paddington, ‘EUS’ for London Euston.

Some ticket purchase and journey planning web sites allow users to enter either the CRS codes or the full name of the location; the ‘short cut’ to the CRS code being favoured by frequent travellers who become familiar with the codes. It is up to the Front End Application designer whether the front end implements this feature.

Journey Planning requests are usually made on the basis that the passenger is interested in planning the fastest journey between origin and destination. There is an option provided that allows the user to specify that they wish to consider direct trains only, to avoid changing trains.

The Journey Planning Call 1 is a simple request – response mechanism. Depending on the input this call also returns journeys in the past, it is up to the FEA to decide whether to allow these.

Railway locations are defined using 3 character 'CRS' and 4 character 'NLC' codes, and the CBE uses these codes in the web service calls. There is a requirement that the FEA is able to translate to and from these codes such that meaningful location names are presented to the customer. The necessary reference data is made available to the FEA by the CBE through separate web service calls.

PROCESS DESCRIPTION

This call from the FEA to the CBE results in a call to IPTIS for route and timetable information. The call is typically the first step in a purchase ticket request process where an end user is more interested in the possible journey times than the actual cost of the fare. The FEA may opt to directly call FEWS 2.1 Journey Planning call 2, which returns the route and timetable data along with the fares data. However, this will result in the fares data being sent back in each response thus requiring the FEA to handle a greater amount of data, which may simply be discarded as the end user refines their journey search.

The option to ‘avoid’ or go ‘via’ a particular location is included e.g. Manchester to Cambridge via Ely will return one set of services if ‘via Ely’ is requested, if it is not requested or set to avoid Ely then other routes via London will be returned.

The input also includes maximum journeys to allow the user to request a number of journeys to be returned, which cannot be greater than the CBE upper limit for maximum journeys.

The journey planning response provided to the FEA contains the details of the journey times including details of the individual legs of the journey where multiple trains are required (trips) and/or transfers and the passenger is required to change between connecting services and/or modes of transport. Transfers are where a non National Rail mode of transport is used to transit between stations, whether this is tube, bus, ferry or walking. For example, a journey planning response for a journey from Manchester to Cambridge via London will include 3 legs:

  1. Manchester – London Euston (Trip)

  2. Transfer by tube to Kings Cross

  3. Kings Cross to Cambridge (Trip).

Data also returned includes the TOC operator of each leg of the service. The output has also been extended to include Train Seating Class/ Train Sleeping Class details, information relating to disruptions affecting journeys returned. Train facilities are returned where available in IPTIS for a journey and also overtaken trains are indicated. Where the journey time is current and the ShowLiveRunning flag is set Real-time travel information (RTTI) where available may be returned for the journey.

ASSUMPTIONS

  1. For Mobile channels, the capability of the receiving device will be a function of the Web Server and not the CBE.
  2. It is assumed that responsibility for continuity of services displayed across successive “Earlier” or “Later” timebands requests lies with the Front End.
  3. It is assumed that all input is validated at source by the FEA on data capture, by means of drop down pick lists and field validation as appropriate to the data content and UI.
  4. For destinations to London Terminals (NLC1072), RSP are providing a data feed via IPTIS to clarify the rules which limit the number of London stations depending on the origin location. . This information will be supplied to the FEA as one of the reference data feeds to the FEA. It is therefore assumed that the Front End Application will need to apply these restrictions as required
  5. The FEA is expected to provide default values for the various flags otherwise the CBE will reject the call with an error message.

VALIDATION

The following validation checks are made by the CBE on the data provided by the FEA. Where the request message from the FEA fails one or more of the validation checks listed below, the CBE will respond with an appropriate error messages as shown in the appropriate API guide.

  1. Dates are correctly formatted
  2. Route codes are numeric and 5 digit
  3. All mandatory fields are present
  4. Station Locations are valid
  5. All Station Locations must be different
  6. Travel Dates < n weeks from current date (n= booking window currently 13 weeks)
  7. Travel (Outbound) date+ time< Return date+ time where there is a return leg
  8. Outbound time can be empty. CBE will use current date time in this case.
  9. Outbound time can be in the past (maximum of 48 hours but configurable ) based on FEA requirements to display recent journeys.
  10. Total journeys returned should be < Maximum Journeys or CBE Configured limit whichever is smaller and should always be > 0.
  11. Max No of Changes must be numeric of (0-9) where 9 means “Any” – if not provided the default value of 9 will be used
  12. If AvoidStationLocation provided but type not specified – the default value of “notvia” will be used
  13. If EnforceRouteVia provided but type not specified – the default value of “passing” will be used
  14. If EnforceRouteVia type=”Route” then only one single 5 digit route code can be specified. No CRS code or NLC code can be specified
  15. If EnforceRouteVia type = changeat or callat then either (CRS and or NLC) must be provided
  16. Service Provider must be a valid two character service provider (as retuned in FEWS 10.5 Get Service Providers).
  17. If ShowOvertaken flag not provided, default to true
  18. If ShowLiveRunning Flag not provided, default to false
  19. If ReduceLondonTransferTime Flag not provided, default to false
  20. If InterchangeTimeIncrease provided value must be an integer between 1 and 120
  21. CheckReservations Flag is False if RC = On Board
  22. InterchangeTimeIncrease and ReduceLondonTransferTime are mutually exclusive – i.e. both cannot be specified in the same input request
  23. Travel Date must be less than or equal to the Extended Horizon End Date
  24. Travel Date must not be an Excluded or Restricted date if for an Extended Horizon booking
  25. Origin and Destination must be a valid pair for an Extended Horizon booking.
  26. If Service Provider Filter is provided but type is not specified, default value is AtLeastOne.
  27. If ShowRepairedJourneys Flag not provided, default to false.
